## ADDED Requirements

### Requirement: ESLint настроен для TypeScript-кода
Проект SHALL содержать конфигурацию ESLint с поддержкой TypeScript (`@typescript-eslint`), применимую ко всем `.ts`-файлам проекта.

#### Scenario: Команда lint выполняется успешно
- **WHEN** разработчик выполняет `npm run lint`
- **THEN** ESLint проверяет все `.ts`-файлы и возвращает код выхода 0 (нет ошибок в шаблоне)

#### Scenario: Нарушение правил обнаруживается
- **WHEN** разработчик добавляет код с явным нарушением правил ESLint (например, `var x = 1`)
- **THEN** `npm run lint` завершается с ненулевым кодом и описанием ошибки

### Requirement: Prettier настроен для форматирования кода
Проект SHALL содержать конфигурацию Prettier (`.prettierrc`) и скрипт `format`, применяющий форматирование ко всем файлам проекта.

#### Scenario: Команда format применяет форматирование
- **WHEN** разработчик выполняет `npm run format`
- **THEN** Prettier форматирует все `.ts`, `.html`, `.css` файлы в соответствии с конфигурацией

#### Scenario: Форматирование идемпотентно
- **WHEN** разработчик выполняет `npm run format` дважды подряд
- **THEN** второй запуск не вносит изменений в файлы

### Requirement: Скрипты npm настроены
`package.json` SHALL содержать скрипты: `dev` (запуск dev-сервера), `build` (сборка), `lint` (проверка ESLint), `format` (форматирование Prettier), `i18n:extract` (синхронизация ключей игровой локали и UI, см. ниже), опционально `i18n:extract-game` и `i18n:extract-ui` по отдельности.

#### Scenario: Все стандартные скрипты присутствуют
- **WHEN** разработчик открывает `package.json`
- **THEN** в секции `scripts` присутствуют ключи `dev`, `build`, `lint`, `format`, `i18n:extract`

### Requirement: Извлечение строк для перевода (i18n:extract)

Проект SHALL содержать реализацию извлечения игровых строк в виде Node-скрипта `scripts/extract-game-i18n.mjs`. npm-скрипт `i18n:extract` SHALL вызывать этот скрипт **и** скрипт синхронизации UI (`extract-ui-i18n.mjs` или эквивалент) **последовательно**. Скрипт `extract-game-i18n.mjs` SHALL обходить файлы `*.ts` под `src/`, **исключая** каталог `src/ifKit/`, находить конструкции с теми же правилами ключа, что и `t()` (включая `t`, `P`, `H1`–`H3`, `em`, `strong`, вызовы вида `` act`…` `(cb)`` с учётом AST TypeScript: у `CallExpression` используется поле `expression`, а не `callee`, и `` goto(…)`…` ``), и **обновлять** JSON-файл **`src/locales/template.game.json`**: для ключей, уже присутствующих в файле, **сохранять** значения; для новых ключей — добавлять пустую строку `""`. Скрипт **не** выполняет машинный перевод. Ключи, присутствующие в JSON, но отсутствующие в коде, MAY сохраняться с предупреждением в консоли (чтобы не терять перевод до ручной очистки).

Команды `build` и `build:tauri-web` SHALL вызывать `i18n:extract` **перед** `tsc` и `vite build`, если иное явно не оговорено отдельным change.

#### Scenario: Скрипт присутствует и синхронизирует ключи
- **WHEN** разработчик выполняет `npm run i18n:extract`
- **THEN** скрипт завершается с кодом 0 и обновляет `src/locales/template.game.json` в соответствии с текущими `` t`…` `` / эквивалентами в коде автора

#### Scenario: Сборка запускает извлечение
- **WHEN** разработчик выполняет `npm run build`
- **THEN** перед компиляцией TypeScript выполняется `i18n:extract` (или эквивалентная команда в той же цепочке)

#### Scenario: Tauri frontend-сборка запускает извлечение
- **WHEN** разработчик выполняет `npm run build:tauri-web`
- **THEN** перед `tsc` и сборкой Vite для Tauri выполняется `i18n:extract` (согласованно с `build-config`)

### Requirement: Синхронизация `*.ui.json` с каноном `UiKey`

Проект SHALL содержать скрипт (отдельный или часть `i18n:extract`), обновляющий файлы `src/locales/<code>.ui.json` **в соответствии** с полным множеством `UiKey`: для **отсутствующих** в JSON ключей добавляются записи (например `""`), **существующие** переводы сохраняются. Машинный перевод **не** выполняется. Для каждого `*.game.json` в каталоге `locales`, включая **`template.game.json`**, SHALL обновляться соответствующий `<code>.ui.json`.

#### Scenario: Новый `UiKey` в движке

- **WHEN** в `ui-keys` добавлен новый `UiKey` и запущен скрипт
- **THEN** **каждый** `*.ui.json` получает этот ключ без потери остального содержимого

#### Scenario: Сосуществование с `i18n:extract` для игры

- **WHEN** разработчик запускает согласованную команду из `package.json` (рекомендуемо: один `i18n:extract`, внутри — и `game`, и `ui`)
- **THEN** `extract-game-i18n.mjs` **по-прежнему** обновляет **только** `*.game.json` по правилам AST для `t` / сцен, а логика `*.ui.json` **не** перетирает `*.game.json` и наоборот

#### Scenario: Создание новой пары

- **WHEN** в проект добавлен новый `xx.game.json`
- **THEN** в документации или скрипте описан способ появления `xx.ui.json` (ручной копией шаблона или тем же extract’ом, если соглашение это поддерживает)

#### Scenario: Шаблон UI синхронизируется вместе с остальными

- **WHEN** в каталоге локалей есть `template.game.json` и запущен скрипт синхронизации UI
- **THEN** файл `template.ui.json` создан или обновлён по канону `UiKey` без потери существующих значений
