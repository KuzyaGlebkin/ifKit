## ADDED Requirements

### Requirement: Источник стабильных ключей UI для `u()`

Проект SHALL трактовать перечисление `UiKey` (экспорт `UI` / тип `UiKey` из `src/ifKit/ui-keys.ts`) как **канонический** список идентификаторов системных строк для `u()`. Документация для автора SHALL указывать на этот модуль (или сгенерированный на его основе перечень) как на ответ на вопрос «какие бывают ключи navbar и модалок».

#### Scenario: Ключ панели инструментов адресуется стабильно

- **WHEN** в коде движка используется `u(UI.toolbarSettings)` (или эквивалентная строка ключа, совпадающая с `UiKey`)
- **THEN** семантика строки определяется сопоставлением с таблицей `UiKey`, а **не** с произвольной игровой фразой из `t`

### Requirement: Слияние встроенного и авторского UI

Документация SHALL объяснять, что во встроенном модуле движка (`BUILTIN_UI`) **содержится ровно один** набор строк — для **языка оригинала игры** (тот же код, что `sourceLanguage` в `defineGame`), чтобы до появления авторских `*.ui.json` chrome отображался на языке источника. Для **каждого** кода языка в `locales` движок **сначала** наполняет слой `u` из `BUILTIN_UI[code]`, **если** этот ключ во встроенной таблице определён; иначе базовый слой для merge **пустой**, далее накладываются шаблон и `locales[code].ui` по правилам движка (**не** подменяя базу встроенным вторым языком из TypeScript, например бывшим `BUILTIN_UI.en`). **Затем** значениями из `locales[code].ui` (если `LocaleBundle` с полем `ui` передан) перекрываются только **непустые** строки. Языки, отличные от оригинала (например `en`), SHALL получать переводы **из JSON** (`<code>.ui.json`) после merge с шаблоном.

#### Scenario: Автор переписывает одну кнопку

- **WHEN** для языка `de` в `defineGame` задан `de: { game: {…}, ui: { "ui.settingsTitle": "…" } }` (или тот же ключ, что в `UI.settingsTitle`)
- **THEN** в интерфейсе на этом языке для заголовка настроек используется **авторское** значение, даже если в `BUILTIN_UI` есть встроенная строка **для** языка оригинала

#### Scenario: Английский chrome берётся из локали, а не из второго слоя в TS

- **WHEN** `sourceLanguage` — `'ru'`, в `BUILTIN_UI` есть только `ru`, а `locales` содержат полностью заполненный `en.ui`
- **THEN** при активном языке `en` вызовы `u` отражают значения из `locales.en.ui`, **а не** отдельный встроенный слой `en` в `builtin-ui.ts`

### Requirement: Парные файлы `*.game.json` и `*.ui.json`

Шаблон и документация SHALL закреплять соглашение: для каждого кода `code`, под которым автор хранит `src/locales/<code>.game.json`, **рядом** в том же каталоге SHALL существовать `src/locales/<code>.ui.json` с картой «ключ `UiKey` — строка» для передачи в `locales[code].ui`, если автор ожидает локализованный chrome для этого кода. Исключения (только `game` в JSON, без `ui`) SHALL быть **осознанными** и задокументированы как сценарий с fallback по правилам `u()` (в т.ч. на данные другой локали в `locales` и/или встроенный язык оригинала в `BUILTIN_UI`).

#### Scenario: Переводчик открывает одну папку

- **WHEN** переводчик ищет все строки для кода `pl`
- **THEN** в `src/locales/` рядом лежат `pl.game.json` и `pl.ui.json` с непересекающимся смысловым назначением (игра vs chrome)

#### Scenario: Новый язык

- **WHEN** автор добавляет новый `xx.game.json`
- **THEN** в процессе/доке предусмотрено добавление `xx.ui.json` (вручную или через `dev-tooling`), иначе **известен** риск английского UI при локализованной игре

### Requirement: UI-ключи стартового меню и логотипа

Перечисление `UiKey` SHALL быть расширено стабильными ключами для: кнопки «Новая игра», «Загрузить» (в контексте меню, если отличается от тулбара), подписи интерактивного логотипа/главного меню, при необходимости — заголовка экрана меню. В `BUILTIN_UI` и шаблоне `template.ui.json` SHALL присутствовать согласованное заполнение этих ключей для языка оригинала и путей перевода.

#### Scenario: Ключи присутствуют в ui-keys

- **WHEN** разработчик открывает `src/ifKit/ui-keys.ts`
- **THEN** перечислены идентификаторы для строк стартового меню, используемые через `u()`
