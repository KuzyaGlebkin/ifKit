## ADDED Requirements

### Requirement: Тип Locales — словарь переводов
Движок SHALL определять тип `Locales = Record<string, Record<string, string>>`, где внешний ключ — код языка (например `'en'`, `'ru'`), внутренний ключ — оригинальная строка с позиционными плейсхолдерами (`{0}`, `{1}` и т.д.), значение — переведённая строка с теми же плейсхолдерами.

#### Scenario: Тип Locales принимает корректный словарь
- **WHEN** автор передаёт `locales: { en: { "Лес": "Forest" } }`
- **THEN** TypeScript принимает значение без ошибок

### Requirement: initI18n инициализирует модуль
Функция `initI18n(locales: Locales, language: string)` SHALL сохранять словарь и активный язык в module-level состоянии. Если `language` пустая строка — определять язык из `navigator.language` (первый сегмент до `-`); если этот код отсутствует в `locales` — брать первый доступный ключ из `locales`; если `locales` пуст — язык остаётся пустым строкой.

#### Scenario: Пустой language — автоопределение из navigator.language
- **WHEN** `initI18n({ en: {...}, ru: {...} }, '')` и `navigator.language === 'en-US'`
- **THEN** активный язык `'en'`

#### Scenario: navigator.language не совпадает ни с одним кодом — первый доступный
- **WHEN** `initI18n({ ru: {...} }, '')` и `navigator.language === 'zh-CN'`
- **THEN** активный язык `'ru'`

#### Scenario: Явный язык устанавливается без автоопределения
- **WHEN** `initI18n({ ru: {...}, en: {...} }, 'ru')`
- **THEN** активный язык `'ru'` независимо от `navigator.language`

### Requirement: Tagged template literal t — перевод строки
Функция `t` SHALL быть tagged template literal функцией: принимает `TemplateStringsArray` и `...unknown[]`, строит ключ вида `"часть0{0}часть1{1}..."`, ищет его в активной локали, подставляет runtime-значения в плейсхолдеры результата. Если ключ не найден — возвращает оригинальную строку с подставленными значениями (fallback).

#### Scenario: Строка без переменных переводится
- **WHEN** активный язык `'en'`, локаль содержит `{ "Лес": "Forest" }`, вызов `` t`Лес` ``
- **THEN** результат `"Forest"`

#### Scenario: Строка с переменными переводится с подстановкой
- **WHEN** активный язык `'en'`, локаль содержит `{ "Найдено {0} золота.": "Found {0} gold." }`, вызов `` t`Найдено ${42} золота.` ``
- **THEN** результат `"Found 42 gold."`

#### Scenario: Ключ не найден — возвращается оригинал с подстановкой
- **WHEN** активный язык `'en'`, ключ отсутствует в локали, вызов `` t`Лес ${name}` ``
- **THEN** результат `"Лес Alice"` (оригинал с подставленным значением)

#### Scenario: locales не переданы — t возвращает оригинал
- **WHEN** `defineGame()` вызван без `locales`, вызов `` t`Привет` ``
- **THEN** результат `"Привет"`

### Requirement: setLanguage меняет активный язык
Функция `setLanguage(lang: string)` SHALL обновлять module-level активный язык. После вызова все последующие вызовы `t` используют новый язык.

#### Scenario: Смена языка применяется к следующим вызовам t
- **WHEN** `setLanguage('en')` вызван, затем `` t`Лес` ``
- **THEN** результат — перевод из локали `'en'`

### Requirement: getAvailableLanguages возвращает список кодов
Функция `getAvailableLanguages(): string[]` SHALL возвращать массив кодов языков из переданного `locales`. Если `locales` не переданы — пустой массив.

#### Scenario: Возвращает коды из переданного словаря
- **WHEN** `locales: { ru: {...}, en: {...} }` переданы
- **THEN** `getAvailableLanguages()` возвращает `['ru', 'en']` (порядок соответствует порядку ключей)

#### Scenario: Без locales — пустой массив
- **WHEN** `defineGame()` без `locales`
- **THEN** `getAvailableLanguages()` возвращает `[]`

### Requirement: Смена языка из настроек с `language: ''`
Для выбора **«Авто»** в настройках движок SHALL обеспечивать **эквивалент** вызова `initI18n(locales, '')` (или **рефакторинг** с общей функцией разрешения, например `setLanguageOrAuto`) **после** сохранения `language: ''`, чтобы `_lang` **не** оставлялся пустым при непустых `locales`. Поведение `setLanguage('')` без resolve SHALL NOT приводить к `t` на пустом `_lang`, если `locales` **не** пусты.

#### Scenario: После «Авто» t работает
- **WHEN** `locales` **не** пусты, пользователь выбрал **«Авто»** (`''` в хранилище)
- **THEN** `t` использует язык по правилу `initI18n(..., '')` (навигатор/первый код), **не** «сыплет» сырые ключи из-за пустого внутреннего языка

#### Scenario: Явный язык с `setLanguage` **не** срезан
- **WHEN** пользователь выбрал `en` и `setLanguage('en')` / `initI18n` с `'en'`
- **THEN** `t` использует `'en'` (как сейчас)

### Requirement: Доступ к словарю при повторной инициализации
Модуль i18n SHALL **хранить** `locales`, переданные в `initI18n`, **для** **повторного** разрешения `''` **без** **повторного** `defineGame`.

#### Scenario: Повторный вызов
- **WHEN** `initI18n` **уже** был, затем **из** модалки `language: ''` **и** re-resolve
- **THEN** **не** требуется рестарт `defineGame`, используется **тот** же `locales`

### Requirement: Словарь системного (UI) текста в локалях

Для каждого кода языка в `locales` движок SHALL поддерживать **отдельный** слой переводов **системного интерфейса** (кнопки панели, модалки, подписи не из сцен), адресуемый **стабильными ключами** (строка идентификатора, не игровой «оригинал»), отличный от плоского слоя строк для `t` по **игровому** оригиналу. Структура в конфиге SHALL быть согласована с `define-game-api` (вложенный `ui` или эквивалент после нормализации).

#### Scenario: Слой UI отделён от игровых строк
- **WHEN** автор задал для языка и `ui`, и `game` (или эквивалент после нормализации)
- **THEN** подпись системной кнопки с ключом `K` **не** зависит от случайного совпадения с игровой фразой в `t` и **берётся** из `ui[K]` (с fallback по правилам движка)

### Requirement: Функция выдачи системной строки по ключу

Движок SHALL предоставлять API `u(key: string): string` для получения **переведённой** системной строки в соответствии с **текущим resolved** языком, совпадающим с логикой `t` / `setLanguageOrAuto` (включая режим `language: ''`).

#### Scenario: Смена языка влияет на u
- **WHEN** игрок сменил язык на `'en'` (или auto разрешилось в `'en'`)
- **THEN** последующий вызов `u` для одного и того же ключа возвращает значение из локали `'en'`

#### Scenario: Ключ отсутствует в слое активной локали, но есть fallback
- **WHEN** у активной локали **нет** значения для ключа в смерженном слое `u`, но значение **есть** по правилам fallback движка (в т.ч. встроенный `BUILTIN_UI` **только** для языка оригинала и/или из другой объявленной локали `locales`, например `en` в данных)
- **THEN** `u` возвращает **это** значение, без пустой строки

### Requirement: Атрибут lang на корне документа

Движок SHALL устанавливать `document.documentElement.lang` на **primary language subtag** (например `en`, `ru`), **совпадающий** с resolved активным языком i18n. Значение SHALL обновляться при инициализации i18n и при каждом изменении resolved-языка. `applySettings` SHALL обеспечивать согласованность через `syncRootLangFromI18n` после инициализации i18n.

#### Scenario: Смена языка в настройках обновляет lang
- **WHEN** пользователь в модалке выбирает другой язык и игра перерисовывается
- **THEN** `document.documentElement.lang` **совпадает** с выбранным (или auto-разрешённым) языком, чтобы вспомогательные технологии объявляли тот же язык, что и UI

#### Scenario: Старт приложения
- **WHEN** `initI18n` завершил разрешение начального языка
- **THEN** `document.documentElement.lang` **не** остаётся в конфликте с фактическим resolved языком (в т.ч. после `applySettings` на старте)

### Requirement: Системные ключи для чекбоксов отключения музыки и звуков
Движок SHALL определять стабильные ключи системного UI (в том же пространстве, что и прочие ключи `UI` для модалки настроек) для подписей или кратких пояснений чекбоксов **включения** музыки и звуков: «включён» чекбокс соответствует **не**заглушенной ветке. Для каждого ключа SHALL быть **непустой** fallback для **языка оригинала** во встроенном `BUILTIN_UI` и, для прочих языков, через заполнение `locales[code].ui` (или эквивалент после merge), согласованный с `u(key)`. Формулировки SHALL не вводить в заблуждение «выключить = галочка стоит»; допускаются краткие нейтральные подписи (включение / звук) или явное **включение**, согласованные с `settings-modal`.

#### Scenario: Ключ чекбокса музыки разрешён через u
- **WHEN** вызывается `u` для ключа пояснения чекбокса музыки
- **THEN** возвращается непустая строка (из локали автора или fallback по правилам движка)

#### Scenario: Ключ чекбокса звуков разрешён через u
- **WHEN** вызывается `u` для ключа пояснения чекбокса звуков
- **THEN** возвращается непустая строка (из локали автора или fallback по правилам движка)

### Requirement: Системные ключи для тихой музыки со скринридерами

Движок SHALL определять стабильные ключи системного UI в пространстве `u()` для подписи переключателя `quietMusicForScreenReader` (и при необходимости отдельный ключ краткого пояснения для вспомогательных технологий). Для каждого ключа SHALL быть **непустой** fallback для **языка оригинала** во встроенном `BUILTIN_UI` и, для остальных языков, через `locales[code].ui`, согласованный с `u(key)` и `settings-modal`.

#### Scenario: Ключ подписи переключателя разрешён через u

- **WHEN** вызывается `u` для ключа подписи переключателя тихой музыки для скринридера
- **THEN** возвращается непустая строка (из локали автора или fallback по правилам движка)

### Requirement: Системные ключи для объявляемой громкости ползунков
Движок SHALL определять стабильные ключи `UI` для строки, **озвучиваемой** вспомогательными технологиями для текущего уровня громкости (музыка и звуки) и, при едином шаблоне, для **той же** подписи рядом с ползунком, с **плейсхолдером** для целого числа 0–100, чтобы автор локали мог задать порядок «число + проценты» в соответствии с языком. Для ключа SHALL быть **непустой** fallback для **языка оригинала** в `BUILTIN_UI` и, для остальных языков, через `locales[code].ui`, согласованный с `u(key)`.

#### Scenario: Ключ уровня громкости разрешён
- **WHEN** шаблон с `{0}` подставляется в строку из `u` по ключу уровня громкости
- **THEN** возвращается непустая локализованная строка, содержащая **целое** процентное значение, а не сырую десятичную долю 0.0–1.0

### Requirement: Сценовая HTML-разметка с переводом (tagged P/H1–H3, em, strong, Phtml)

Движок SHALL экспортировать из `src/ifKit` (через `index.ts`) **tagged template literal** функции `P`, `H1`, `H2`, `H3`, которые **сначала** применяют к шаблону ту же логику, что и `t`, **затем** вставляют разметку в HTML-буфер сцены (эквивалент `p`/`h1`–`h3` + `appendHtml`).

Движок SHALL экспортировать **tagged** `em` и `strong`, возвращающие **строку** HTML-фрагмента (внутренняя разметка + `t`).

Движок SHALL экспортировать `Phtml(innerHtml: string, attrs?)` для одного `<p>` по **уже собранной** строке (в т.ч. склейка нескольких вызовов `t`).

#### Scenario: Абзац и плейсхолдеры согласованы с t

- **WHEN** вызывается `` P`…` `` / `` H1`…` `` / `` em`…` `` / `` strong`…` `` при тех же `locales`, что и для эквивалентного `` t`…` ``
- **THEN** фрагмент в буфере/DOM согласован с результатом `t` плюс обёртка тега

#### Scenario: Phtml для склейки

- **WHEN** автор передаёт в `Phtml` строку, уже прошедшую `t` и/или конкатенацию
- **THEN** в сцене появляется один блок `<p>` с этой разметкой

### Requirement: **BREAKING** — публичный string-API старых тегов сцены

Публичный экспорт `src/ifKit` SHALL **не** предлагать приём **готовой** `string` для `P`/`H1`/`H2`/`H3`/`em`/`strong` / `Em`/`Strong` **в прежней форме** (только `string` без tagged `t` в момент вставки исходного текста), совместимой со старой схемой `P`/`H1` + аргумент-строка. Миграция на tagged API или `Phtml` обязательна.

#### Scenario: Старый стиль `P(строка)` не поддерживается

- **WHEN** автор пишет вызов, несовместимый с новыми сигнатурами
- **THEN** TypeScript/сборка требуют миграции (или иной явной ошибки контракта)

### Requirement: Подписи `act` и `goto` через t

`SceneContext.act` SHALL принимать tagged-шаблон, строить подпись кнопки через `t` (тот же слой `locales`, что и `t`), и возвращать функцию `(cb) => void`.

`SceneContext.goto` SHALL иметь форму: сначала ключ целевой сцены, затем tagged-шаблон для подписи перехода; внутри подписи SHALL применяться `t` так же, как для `t` в сцене.

#### Scenario: act с шаблоном

- **WHEN** сцена вызывает `` act`Подпись`(state => { … }) ``
- **THEN** кнопка действия отображается с тем же переводом, что дал бы `` t`Подпись` ``

#### Scenario: goto с шаблоном

- **WHEN** сцена вызывает `` goto('key')`Подпись` ``
- **THEN** кнопка перехода отображается с тем же переводом, что дал бы `` t`Подпись` ``

### Requirement: Читаемый web-артефакт поддерживает локализацию без репозитория

При соблюдении требования `build-config` о **не**минифицированной web single-file сборке, итоговый `dist/<name>_<version>.html` SHALL оставаться достаточно читаемым, чтобы автор или переводчик **без** доступа к исходному репозиторию мог:

- найти в файле вложенный объект `locales` (или эквивалентную структуру после сборки) и пары «ключ `t` / значение перевода»;
- при необходимости **вручную** править строки перевода внутри инлайн-скрипта и сохранять файл для локальной проверки (валидность JS и сохранение синтаксиса — ответственность редактора).

Это **не** заменяет хранение канонических переводов в исходниках игры; это контракт **удобства** дистрибутива для обходных сценариев (срочная правка, вытягивание шаблона для внешнего переводчика).

#### Scenario: Поиск известного ключа локали в артефакте
- **WHEN** в `defineGame` передан `locales` с записью `"Лес": "Forest"` и выполнена web-сборка по `build-config` без минификации
- **THEN** в `dist/<name>_<version>.html` текстовый поиск находит подстроку `Лес` и/или `Forest` в контексте объекта переводов (а не только в сжатом недифференцируемом виде)

### Requirement: Канон ключей в репозитории и извлечение без машинного перевода

Список ключей игровых строк (тот же алгоритм склейки, что у `t`: литералы + `{0}`, `{1}`, …) SHALL поддерживаться в актуальном состоянии относительно кода автора через скрипт `i18n:extract` (`dev-tooling`), в т.ч. в `template.game.json` и пер-языковых `*.game.json` (`define-game-api`, `locale-templates`). Переводы в JSON SHALL вноситься **вручную** или внешними инструментами по желанию автора; движок и скрипт извлечения **не** обязаны и **не** гарантируют автоматический перевод.

#### Scenario: Новая строка в сцене требует ключа в JSON
- **WHEN** автор добавляет в код новый вызов `` t`…` `` (или эквивалент с тем же правилом ключа) и выполняет `i18n:extract` / `npm run build`
- **THEN** в JSON локали появляется новый ключ с пустым или сохранённым значением согласно правилам `dev-tooling`, а отсутствие заполненного перевода проявляется как fallback `t` на исходный язык (существующее поведение)

### Requirement: Ожидаемое заполнение UI для языка без встроенного `BUILTIN_UI`

Документация к движку SHALL явно указывать: для кода `code` присутствующего в `locales`, если **`BUILTIN_UI[code]`** не задан (типично для **всех** языков, кроме оригинала), базовый merge **не** копирует второй встроенный язык из TypeScript; chrome для `code` SHALL строиться из шаблона и `locales[code].ui`, а пропуски SHALL заполняться по **документированной** цепочке fallback (в т.ч. из другой локали в `locales`, например `en`, и/или встроенного языка оригинала), **а не** из несуществующего `BUILTIN_UI.en` в коде.

#### Scenario: Язык с полным `*.ui.json`

- **WHEN** для `code` нет `BUILTIN_UI[code]`, но `locales[code].ui` содержит **непустые** значения для всех `UiKey`, используемых текущим UI
- **THEN** `u` **не** откатывается к устаревшему «встроенному английскому из TS» **для** этих ключей

#### Scenario: Частично заполненный `ui`

- **WHEN** для ключа `UiKey` в `locales[code].ui` значение пропущено или пустое, а `BUILTIN_UI[code]` отсутствует
- **THEN** `u` для этого ключа использует значение по правилам merge и fallback (в т.ч. из языка оригинала в `BUILTIN_UI` и/или из данных другой локали автора), **без** восстановления отдельного типизированного слоя `en` в `builtin-ui.ts`

### Requirement: Пустая строка в игровой локали для `t`

Функция `t` SHALL трактовать значение `""` для ключа в активной игровой локали как отсутствие перевода (то же поведение, что при отсутствии ключа): подставлять значения в шаблон оригинальной строки-ключа и не возвращать «пустой» перевод.

#### Scenario: Новый ключ из extract с пустым значением

- **WHEN** активный язык `'en'`, в локали `{ "Лес": "" }`, вызов `` t`Лес` ``
- **THEN** результат `"Лес"` (оригинал с подстановками, как при отсутствии перевода)

#### Scenario: Непустой перевод сохраняется

- **WHEN** в локали `{ "Лес": "Forest" }`, вызов `` t`Лес` ``
- **THEN** результат `"Forest"`

### Requirement: Стабильный ключ UI для пресета акцента `orange`

Движок SHALL определять стабильный ключ системного UI (в пространстве, согласованном с `ui-keys` / `define-game-api`, например `ui.settingsAccentOrange`) для подписи сегмента пресета `orange` в ряду «Акцент» модалки настроек, с **встроенным** fallback в `builtin-ui` (или эквивалент) для **языка оригинала** и заполнением остальных языков через `locales[code].ui`, согласованно с `u(key)`. Ключ, соответствующий удалённому пресету `violet` (`ui.settingsAccentViolet`), SHALL NOT оставаться единственным идентификатором для оранжевого пресета.

#### Scenario: `u` возвращает подпись для оранжевого пресета

- **WHEN** вызывается `u` для ключа подписи пресета `orange` и активна локаль с заполненным значением
- **THEN** возвращается непустая строка, семантически соответствующая оранжевому/мягко-оранжевому пресету акцента
