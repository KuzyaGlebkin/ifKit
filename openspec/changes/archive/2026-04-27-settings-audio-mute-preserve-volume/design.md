## Context

В `Settings` уже хранятся `musicVolume` и `soundVolume` (0–1); аудиомодуль выставляет мастер-гейны при инициализации и через `setMusicVolume` / `setSoundVolume`. Модалка настроек показывает два `range` с подписями «Музыка» / «Звуки». Нужны отдельные флаги «полностью выкл.» без изменения чисел на ползунках.

## Goals / Non-Goals

**Goals:**

- Два булева поля в настройках (музыка / звуки), персистентность в `ifkit:settings`, участие в сбросе и импорте/экспорте.
- UI: между текстовой подписью строки и блоком с ползунком — чекбокс; ползунок по-прежнему отражает сохранённую долю громкости.
- При `musicMuted === true` слышимая громкость музыки — 0; при снятии mute — снова согласно `musicVolume` (аналогично звуки).
- Совместимость: отсутствие полей в старых JSON — как `false`.

**Non-Goals:**

- Отдельный «master mute» на всё аудио одной кнопкой.
- Изменение модели кроссфейда или кэша буферов.
- Привязка mute к системному mute ОС.

## Decisions

1. **Имена полей:** `musicMuted` и `soundMuted` (boolean), движковые умолчания `false`. Слияние с `authorDefaults` / валидация при загрузке — как у остальных полей `Settings`.

2. **Аудио: номинал vs эффект:** модуль хранит последние номинальные `musicVolume` / `soundVolume` (как сейчас) и флаги mute (из настроек или зеркально в модуле). Значение `GainNode.gain` для музыки SHALL равняться `musicVolume * (musicMuted ? 0 : 1)` (и аналогично для звуков). Вызов `setMusicVolume(v)` обновляет номинал и **пересчитывает** gain с учётом текущего mute; отдельный путь при смене mute (из UI) делает то же. Так слайдер не трогает поле mute, а mute не перезаписывает `musicVolume` в хранилище.

3. **Инициализация AudioContext:** при первом создании контекста читать полные настройки (или переданные из движка значения) и выставить оба мастер-гейна в **эффективные** значения с учётом mute.

4. **Публичный API аудио:** расширить экспорт функциями вида `setMusicMuted` / `setSoundMuted` **или** одной `applyAudioFromSettings(pick)` — выбрать в реализации минимальный дифф; критерий: движок/UI не дублируют формулу gain вне `audio.ts`.

5. **Разметка модалки:** порядок в строке — `<label>` (или `span` + `label for` только на ползунок, если чекбокс отдельно) для названия, затем `<input type="checkbox">`, затем обёртка с `range` и процентами. Для a11y: у чекбокса явная подпись через `<label>` / `aria-label` с ключами `u(...)`, чтобы не было «голого» переключателя.

**Альтернатива (отклонена):** ставить ползунок в 0 при mute и восстанавливать прежнее значение из скрытого поля — хуже для UX (проценты «прыгают») и для экспорта состояния.

## Risks / Trade-offs

- **[Риск] Расхождение спеки и кода по «заглушкам» в `settings-modal`:** в базовой спеке ещё упоминаются заглушки; дельта приводит требование в соответствие с `user-settings` и реальным аудио. → **Митигация:** MODIFIED-блок в change для модалки.

- **[Риск] Crossfade / per-track гейны:** если часть громкости задаётся не только мастером, нужно убедиться, что mute режет итоговый слышимый уровень. → **Митигация:** mute только на `_masterMusicGain` / `_masterSoundGain` (единая точка).

## Migration Plan

- При `loadSettings`: если в объекте нет `musicMuted` / `soundMuted`, подставить `false`.
- Экспорт/импорт хранилища без миграции файлов — новые ключи опциональны в JSON.
- Откат: удалить поля из кода и игнорировать ключи в старых файлах (как неизвестные) при откате релиза — вне scope этого change.
