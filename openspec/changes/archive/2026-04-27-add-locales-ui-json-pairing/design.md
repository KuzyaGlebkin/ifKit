## Context

`normalizeLocalesInput` остаётся: для каждого `code` в `locales` UI = `BUILTIN_UI[code] ?? BUILTIN_UI.en` + авторский `ui`. Если авторам нужен **новый** язык `de`, `pl`, `by`, …, для которого в `BUILTIN_UI` **нет** отдельного объекта, до заполнения `locales[code].ui` chrome остаётся **английским** из встроенного слоя. Игровой слой при этом справляется через `de.game.json` — асимметрия и путаница. Предлагаемый якорь: **одинаковая дисциплина по файлам** — на каждый `*.game.json` в проекте автора **пара** `*.ui.json`.

## Goals / Non-Goals

**Goals:**

- Сделать **предсказуемый** путь: добавил `<code>.game.json` — рядом лежит или создаётся `<code>.ui.json` с ключами `UiKey`.
- Покрыть **любой** язык в `locales`, а не единичный `by` в `BUILTIN_UI`.
- Автоматизировать **синхронизацию ключей** `*.ui.json` с каноном `ui-keys` (как extract для `t`).

**Non-Goals:**

- Машинный перевод.
- Смена алгоритма merge в `i18n` (если не всплывёт дефект).
- Обязательное наполнение `BUILTIN_UI` новыми нативными языками — **не** требуется, если `*.ui.json` закрывают сценарий.

## Decisions

1. **Соглашение об имённых парах `src/locales/<code>.{game,ui}.json`**

   - **Решение:** в шаблоне для **каждого** кода, для которого есть `game`, добавить (или сгенерировать) `<code>.ui.json`. Именование строго по коду: `en.game.json` ↔ `en.ui.json`, `by.game.json` ↔ `by.ui.json`.
   - **Рационale:** тот же mental model, что у игры; grep и обзор папки не пропустит «второй половины».
   - **Альтернатива:** один `ui` на все языка — хуже для диффов и внешних переводчиков.

2. **`defineGame` и импорты**

   - **Решение:** `locales: { en: { game: enGame, ui: enUi }, by: { game: byGame, ui: byUi }, ... }` с импортом JSON из `src/locales/*.ui.json`. Плоский `locales: { en: { ... } }` остаётся для обратной совместимости, но **шаблон** переводит на `LocaleBundle` с двумя полями, если в проекте больше одного языка / нужен не-en chrome.

3. **Скрипт в `dev-tooling`**

   - **Решение:** отдельный шаг, например `i18n:extract-ui` (или `i18n:extract` вызывает оба), который читает `UiKey` (через парс `ui-keys.ts` или константный список), обновляет **каждый** `*.ui.json` в `src/locales/`: существующие значения сохраняются, новые ключи — `""` или плейсхолдер, как в `extract-game-i18n.mjs`.
   - **Рационale:** дублирует дисциплину `i18n:extract` и снимает вопрос «где взять список ключей».

4. **Роль `BUILTIN_UI`**

   - **Решение:** оставить **узкий** встроенный набор (например `en`, `ru`) как дефолт движка; **не** пытаться покрывать в `builtin-ui` все языки игр. Полноту UI для `code` **без** `BUILTIN_UI[code]` несут `*.ui.json`. Док: при пустом/частичном `*.ui.json` пустоты, как сейчас, добираются из `en` в merge.

5. **Документация**

   - **Решение:** один раздел: «два JSON на язык», таблица «слой / файл / API», ссылка на `ui-keys` и порядок merge.

## Risks / Trade-offs

- **[Риск] Два файла вместо одного** — **Митигация:** снимается путаницей `t` vs `u` и сопровождается extract’ом; дублирования строк игры в UI нет (разные пространства ключей).
- **[Риск] Рассинхрон `UiKey` и JSON** — **Митигация:** обязательный `extract-ui` в цепочке `build` рядом с `i18n:extract`.
- **[Риск] Миграция существующих проектов** — **Митигация:** `extract-ui` создаёт пустой `en.ui.json` + для прочих кодов из `locales`; **BREAKING** не в API, только в соглашении шаблона.

## Migration Plan

- Добавить `*.ui.json` и правку `game.ts` в шаблоне; прогнать `i18n:extract-ui` один раз, закоммитить.
- Существующие игры: добавить файлы и `LocaleBundle` **или** оставить как есть (chrome на `en` до появления `ui`).
