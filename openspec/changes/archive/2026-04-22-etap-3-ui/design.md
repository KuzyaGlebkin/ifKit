## Context

Движок ifKit создаёт DOM-структуру через `engine.ts` (`getOrCreate`) и рендерит контент через `innerHTML` в `renderer.ts`. На входе в этап 3:

- Четыре зоны (`#static`, `#scene-content`, `#scene-acts`, `#scene-gotos`) уже существуют в коде, но создаются как безымянные `<div>` без семантики
- Кнопки `act` и `goto` рендерятся одинаково — `<button>` без классов и ARIA-атрибутов
- После каждого вызова `innerHTML = ''` фокус теряется — вспомогательные технологии не получают сигнала об изменении контента
- `layout.ts` — пустой стаб, `style.css` — минимальный
- Публичный API автора (`defineGame`, `SceneContext`) остаётся неизменным

## Goals / Non-Goals

**Goals:**
- Добавить `aria-label` и `role` к зонам интерфейса в `index.html`
- Визуально и семантически разделить кнопки `act` (outlined) и `goto` (filled + `aria-label`)
- После каждой перерисовки перемещать фокус в приоритетную зону: `#static` → `#scene-content` → первая кнопка
- Ввести систему CSS-переменных `--ifk-*` со светлой/тёмной темой и тремя слоями переопределения

**Non-Goals:**
- Реализация сохранения пользовательских настроек (localStorage) — это этап 4
- UI настроек темы для игрока — это этап 4
- Переключение темы через кнопку — только CSS-структура сейчас
- Изменение публичного API (`GameConfig`, `SceneContext`)
- Кастомные компоненты и плагины — это этап 9

## Decisions

### 1. Фокус-менеджмент живёт в `scenes.ts`

`scenes.ts` содержит `runGameLoop` — единственную точку, где происходит перерисовка всех зон. Добавление вызова `focusAfterRender()` после трёх `flush`-вызовов сохраняет ответственность в одном месте и не требует нового модуля.

Альтернатива: вынести в `layout.ts`. Отклонена — `layout.ts` не знает о цикле рендеринга, потребовались бы лишние зависимости.

### 2. Фокус, но не `aria-live`

При перерисовке `#scene-content` — два способа оповестить скринридер:
- `aria-live="polite"` — скринридер читает контент автоматически
- Программный фокус на контейнер (с `tabindex="-1"`) — скринридер читает при получении фокуса

Выбран **только фокус**. При одновременном использовании обоих скринридеры (NVDA, VoiceOver) зачитывают контент дважды. Фокус также даёт игроку контроль над темпом — он сам решает, читать сейчас или пропустить.

### 3. Приоритет фокуса: static → scene-content → первая кнопка

```
function focusAfterRender(staticEl, sceneEl, actsEl, gotosEl):
  1. Если staticEl существует → первый focusable child статичной зоны
                               → иначе сам staticEl (tabindex="-1")
  2. Иначе если sceneEl не пуст → sceneEl (tabindex="-1")
  3. Иначе → первая кнопка в actsEl ?? gotosEl
```

`tabindex="-1"` на `#scene-content` устанавливается один раз в `layout.ts` при инициализации, а не при каждой перерисовке.

### 4. Классы и `aria-label` для кнопок act / goto

`renderer.ts` знает тип кнопки при создании — изменение минимально:

```
act  → btn.className = 'btn-act'
       btn.setAttribute('aria-label', `Act: ${label}`)

goto → btn.className = 'btn-goto'
       btn.setAttribute('aria-label', `Go to: ${label}`)
```

`aria-label` на английском: "Act" и "Go to" — технические роли, аналогично тому, как `role="button"` не переводится. Если в будущем потребуется i18n — добавляется поле `locale` в `GameConfig` (этап 7).

### 5. CSS-переменные `--ifk-*`, три слоя, единый акцент

Префикс `--ifk-` изолирует токены движка от переменных автора игры. Единый `--ifk-color-accent` без пары light/dark — автор и игрок задают одно значение, оно работает в обеих темах. Если потребуется разный акцент для тем — решается через CSS-нативную логику `color-mix()` или `oklch()` в будущем.

Три слоя (каскад):
1. `:root` в `style.css` — дефолты движка
2. Переопределение автора в его `style.css` или `game.ts` (через `<style>`)
3. `document.documentElement.style.setProperty(...)` из JS — наивысший приоритет для пользовательских настроек (этап 4)

## Risks / Trade-offs

- **Двойной фокус в static** — если автор сам вставил focusable элемент первым в static, фокус будет на нём, а не на начале контента сцены. Это ожидаемое поведение: автор управляет структурой static зоны сознательно.
- **`innerHTML` сбрасывает `tabindex`** — `#scene-content` перезаписывается через `innerHTML`. `tabindex="-1"` установлен на самом `<div>`-контейнере, а не на его потомках — проблемы нет.
- **Потеря фокуса при `rerender()`** — `act`-коллбэк вызывает `rerender()` → `runGameLoop()` → перерисовка той же сцены. Фокус снова управляется `focusAfterRender`, что сбросит его на начало — это может раздражать при повторных `act` без перехода. Митигация: принять как ограничение v1, в будущем можно запомнить и восстанавливать позицию фокуса.
- **Контраст акцентного цвета** — единый `--ifk-color-accent` должен давать контраст ≥ 3:1 против фона кнопки (filled) как в светлой, так и в тёмной теме. Дефолтный акцент подбирается с проверкой WCAG AA.
