## Context

- Сегодня `BUILTIN_UI` задаёт **два** полных набора: `ru` и `en`. Пример `defineGame` с `sourceLanguage: 'ru'` подмешивает `locales` через `mergeAuthorLocalesWithTemplate`, затем `normalizeLocalesInput` строит слой `u`: база из `BUILTIN_UI[lang] ?? BUILTIN_UI.en`, поверх — непустые значения из `locales[lang].ui`.
- Цель автора: **одна** ментальная модель — оригинал и движковый UI на языке источника в коде/встроенной таблице; **все** прочие языки, включая английский, в `src/locales/*.ui.json`.

## Goals / Non-Goals

**Goals:**

- В `builtin-ui.ts` экспортировать только слой для **языка оригинала** (`sourceLanguage` автора; в репозитории ifKit пример — `ru`).
- Перенести текущие английские строки из TS в `src/locales/en.ui.json` как непустые значения по стабильным ключам `UiKey`.
- Обновить merge так, чтобы отсутствие `BUILTIN_UI[code]` **не** подменяло базу встроенным английским слоем из TS; английский chrome приходит из данных `locales.en.ui` (после merge с шаблоном).
- Согласовать цепочку fallback в `u()` с новой моделью (без захардкоженного `BUILTIN_UI.en`).

**Non-Goals:**

- Менять модель `t()` / игровые `*.game.json`.
- Расширять список поддерживаемых языков в движке.
- Автоматически генерировать `en.ui.json` из старого TS (ручной перенос по change достаточен).

## Decisions

1. **Ключ `BUILTIN_UI` в TypeScript** — только `sourceLanguage` для данной кодовой базы (`ru` в ifKit). Игры с другим `sourceLanguage` в будущем держат **один** встроенный слой под свой код (замена содержимого `builtin-ui`, не добавление второго полного языка в тот же механизм).

2. **Базовый слой в `normalizeLocalesInput` для `lang`**:  
   - `built = BUILTIN_UI[lang]` если есть;  
   - иначе **не** копировать `BUILTIN_UI.en`; начать с пустого объекта и дальше применить уже смёрженные template + author `ui` (как сейчас для поверхностных ключей).  
   Или эквивалентно: для каждого `lang` ввести `builtInForLang = BUILTIN_UI[lang]` только при наличии ключа в `BUILTIN_UI`.

3. **Fallback в `u()`**: заменить опоры на `BUILTIN_UI.en` на: активная локаль → встроенный слой **только для языка, присутствующего в `BUILTIN_UI` (ожидаемо один — исходный)** → слои других локалей в данных (например `en` в `_ui` после merge) в порядке, согласованном со спеком (минимум: явный список в задачах/тестах), затем сырой ключ.

4. **`mergeAuthorLocalesWithTemplate` / пустые значения**: сохранить правило «`''` в author не перекрывает» для overlay поверх встроенного; после переноса `en` в JSON английский слой не зависит от пустого `en` в TS.

**Alternatives considered**

- **Оставить `en` в TS как «лингва франка»** — отвергнуто: противоречит желаемой модели «все переводы в JSON».
- **Дублировать `ru` и в JSON** — не требуется для этого change: исходный язык остаётся во встроенной таблице.

## Risks / Trade-offs

- **[Risk]** Авторы мини-игр без заполненного `en.ui.json` потеряют английский chrome из коробки. **→ Mitigation:** зафиксировать **BREAKING** в proposal; заполнить `en.ui.json` в образце и указать в tasks проверку.
- **[Risk]** Ошибка в порядке fallback в `u()` даёт сырой ключ или язык оригинала там, где ожидали EN. **→ Mitigation:** ручная проверка UI на `en` и минимум один автоматический/ручной сценарий.
- **Trade-off:** встроенный размер бандла: английский уходит в JSON, который всё равно попадает в бандл вместе с `game.ts`; общий размер близок, меняется только «слой доверия» автора.

## Migration Plan

1. Заполнить `en.ui.json` из текущего `BUILTIN_UI.en`.
2. Удалить `en` из `builtin-ui.ts`.
3. Изменить `normalizeLocalesInput` / `u()`.
4. Собрать и проверить сценарии: язык `ru`, `en`, `by`, авто.
5. Откат: вернуть `BUILTIN_UI.en` и предыдущую merge-логику из git.
