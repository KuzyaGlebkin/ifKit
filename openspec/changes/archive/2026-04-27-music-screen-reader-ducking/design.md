## Context

Фоновая музыка в ifKit идёт через Web Audio (`AudioContext` → `_masterMusicGain`). Речь скринридера (внешнего AT) микшируется ОС с остальным звуком приложения; веб-API не сообщает, когда NVDA/JAWS говорят, поэтому нельзя реализовать классическое автоматическое ducking «на время речи». Пользовательский флаг с постоянным ослаблением только музыки — стандартный практичный способ оставить ползунок громкости осмысленным и при этом не перебивать TTS.

## Goals / Non-Goals

**Goals:**

- Персистентная настройка «тише фоновая музыка при использовании скринридера», влияющая только на ветку музыки (не на SFX).
- Сохранённые `musicVolume` и логика `musicMuted` не меняются: масштабирование — отдельный множитель к **эффективному** gain.
- Значение по умолчанию **выкл.** (поведение как сейчас для существующих сохранений и авторских умолчаний).

**Non-Goals:**

- Определение момента речи внешнего скринридера или синхронизация с `speechSynthesis` (в проекте нет встроенного озвучивания текста через Web Speech в качестве замены AT).
- Изменение системного микшера Windows или отдельного аудиоустройства для AT.
- Автоматическое включение флага по эвристикам (`prefers-reduced-motion` и т.п.) без явного согласия пользователя.

## Decisions

1. **Имя поля в `Settings`:** `quietMusicForScreenReader: boolean`. Явно указывает назначение; default `false`.

2. **Формула эффективной громкости музыки:**  
   `effectiveMusic = musicVolume × (musicMuted ? 0 : 1) × (quietMusicForScreenReader ? SCREEN_READER_MUSIC_ATTENUATION : 1)`  
   Константа `SCREEN_READER_MUSIC_ATTENUATION` — положительное число ≤ 1 (рекомендация в коде: **0.25**): достаточное снижение без полного отключения музыки одним флагом.

3. **API аудиомодуля:** экспорт `setQuietMusicForScreenReader(on: boolean): void` (или эквивалентное имя), вызываемое из пути применения настроек рядом с `setMusicVolume` / `setMusicMuted`; при неинициализированном контексте — сохранение флага и применение при первом `PlayMusic`, аналогично другим полям.

4. **UI:** отдельный элемент управления в секции аудио — после строки «Музыка» (ползунок + mute) логично разместить один чекбокс с `<label>` и при необходимости кратким пояснением (`aria-describedby` или вспомогательный текст), без изменения порядка существующих ползунков Tab внутри строк музыка/звуки: новый фокусируемый элемент включается в общий порядок Tab модалки (конкретный порядок — после строки музыки, до строки звуков — зафиксировать в спеках `settings-modal`).

5. **Миграция:** отсутствие ключа в JSON → `false`. Экспорт/импорт включает поле; сброс к `authorDefaults` восстанавливает значение из `engineDefaults`/автора.

**Альтернативы:** только снижение `engineDefaults.musicVolume` для всех — отклонено: бьёт по обычным пользователям. Временное приглушение по `focus` — хрупко и непредсказуемо для SR.

## Risks / Trade-offs

- [Слишком слабое/сильное приглушение] → Фиксированный коэффициент 0.25 можно вынести в одну константу; при отзывах пользователей — отдельная настройка «уровня приглушения» вынесена за рамки этой задачи.
- [Дублирование логики gain] → Единая функция `effectiveMusicGain()` в `audio.ts`, как сейчас для mute.

## Migration Plan

Поле добавляется с default `false`; существующие установки не меняют слышимость до явного включения в настройках. Откат: удалить поле и ветку в audio (версия кода до изменения).
