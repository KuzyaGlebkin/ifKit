## Context

Сегодня Vite включает JSON локалей только если на них есть статический импорт. Авторы добавляют импорты в `game.ts` и передают `locales` в `defineGame`. Скрипт `extract-game-i18n.mjs` пишет ключи в `src/locales/en.game.json`, что для игр с одним языком и неанглийским исходным текстом провоцирует «фиктивный» `en` в `locales` и в настройках. В `i18n.ts` значение `""` в словаре игры для `t` уже сейчас ломает fallback (пустая строка не срабатывает `?? key`).

## Goals / Non-Goals

**Goals:**

- Зафиксировать канонические `src/locales/template.game.json` и `src/locales/template.ui.json` как опору для извлечения ключей и для графа сборки без импортов в авторском `game.ts`.
- Импортировать эти модули из `src/ifKit` и до `normalizeLocalesInput` / `initI18n` подмешивать данные шаблона к **каждому** авторскому языковому слою (`game` и при необходимости `ui`), не добавляя псевдоязык в переключатель.
- Перевести `extract-game-i18n.mjs` на вывод в `template.game.json`; `extract-ui-i18n.mjs` уже обходит все `*.game.json`, поэтому создаст/обновит `template.ui.json` автоматически.
- Исправить семантику `t` для пустых строк перевода в активной локали (как «нет перевода»), чтобы новые ключи из extract не ломали отображение.

**Non-Goals:**

- Машинный перевод или смена формата ключей `t`.
- Удаление поддержки отдельных `en.game.json` / `by.game.json` у авторов — они остаются для реальных языков; образцовая игра может продолжать их использовать.

## Decisions

1. **Имена файлов и расположение** — `template.game.json` / `template.ui.json` под `src/locales/`, как предложено. Код языка `template` в `defineGame` не передаётся: шаблон не является записью в `locales`.

2. **Точка импорта** — новый небольшой модуль в `src/ifKit` (например `locale-templates.ts`), импортирующий JSON и экспортирующий объекты для `engine.ts`. Путь импорта относительно `ifKit` → `../locales/template.game.json` (и `.ui.json`).

3. **Алгоритм merge** — перед `normalizeLocalesInput` движок объединяет для каждого ключа языка из авторского `fullRaw`:
   - `game`: `{ ...templateGame, ...authorGame }` (автор перекрывает шаблон; ключи только из шаблона остаются доступны для `t`).
   - `ui`: аналогично для объектов `ui`, если они есть у автора; пустые строки в авторском `ui` уже игнорируются при merge с `BUILTIN_UI` в `normalizeLocalesInput` (существующее правило).

4. **Пустые значения в `template.game.json`** — после изменения `t` пустая строка эквивалентна отсутствию ключа, поэтому extract может безопасно добавлять `""` для новых ключей.

5. **Образцовый `game.ts`** — убрать импорты, существующие только ради каталога ключей; оставить импорты реальных языков (например `ru`/`by`/`en` по факту проекта). Каталог ключей обслуживает `template.*`.

**Альтернативы:**

- Отдельный «служебный» код языка в `locales`, потом фильтровать в UI — отклонено: риск утечки в настройки и лишняя модель.
- Только side-effect import без merge — отклонено: не решает консистентность ключей в рантайме и не устраняет необходимость дублировать ключи по языкам.

## Risks / Trade-offs

- **[Risk]** Авторы с кастомным путём к локалям ожидают `en.game.json` как цель extract → **Mitigation**: обновить `dev-tooling` и README/шаблон; при отсутствии файла скрипт создаёт `template.game.json`.
- **[Risk]** Слияние шаблона с большим числом языков дублирует ключи в памяти → **Mitigation**: приемлемо для типичных размеров IF; при росте можно оптимизировать отдельным change.
- **[Trade-off]** Поведение `t` меняется для намеренного `""` — крайне редкий случай; задокументировать в дельте `i18n`.

## Migration Plan

1. Добавить `template.*` (можно начать с `{}` или копии ключей из текущего `en.game.json` при первом прогоне extract).
2. Переключить extract на `template.game.json`; запустить `i18n:extract`.
3. Реализовать импорт и merge в ifKit; обновить образцовый `game.ts`.
4. Проверить `build` / `build:tauri-web` и настройки языка (нет лишних кодов).

**Rollback:** вернуть целевой файл в `extract-game-i18n.mjs` и убрать merge-модуль (git revert).

## Open Questions

- Нужно ли в образцовой игре **полностью** убрать отдельный `en` слой, если английский только в `template` — решается при реализации по желанию автора репозитория (spec допускает реальные `*.game.json` параллельно).
