## Context

Текущая реализация управления фокусом вызывает `sceneEl.focus()` после каждого рендера сцены. NVDA и Narrator при получении фокуса элементом с `tabindex="-1"` переключаются из Browse Mode (виртуальный буфер, читает весь текст) в Application Mode (только интерактивные элементы). В результате статичный текст сцены (`<p>`, `<ul>` и т.д.) недоступен скринридеру через Tab.

Текущий `ui-zones` спек явно запрещает `aria-live` на `#scene-content` с формулировкой «чтобы избежать двойного зачитывания» — этот запрет необходимо снять.

## Goals / Non-Goals

**Goals:**
- Контент сцены автоматически зачитывается скринридером при каждом переходе
- Пользователь не застревает в Application Mode — может читать сцену в Browse Mode
- Фокус после рендера остаётся на интерактивном элементе (кнопке), а не на контейнере

**Non-Goals:**
- Roving tabindex по отдельным абзацам — это сверхсложность без реальной пользы для IF-игры
- Поддержка старых скринридеров без поддержки `aria-live`
- Изменение поведения для случаев со `static`-зоной (она сохраняет приоритет фокуса)

## Decisions

### 1. `aria-live="polite"` + `aria-atomic="true"` на `#scene-content`

**Решение**: добавить оба атрибута при инициализации в `engine.ts`.

**Почему `polite`**: не прерывает текущее зачитывание — подходит для сценарного текста, не срочного.

**Почему `atomic`**: `aria-atomic="true"` говорит скринридеру читать весь регион как единицу при изменении, а не только изменённые узлы. Без него скринридер может зачитать только последний добавленный параграф.

**Альтернатива — отдельный `aria-live` announcer**: невидимый `<div aria-live="polite">` в `<body>`, куда копируется текст. Сложнее, требует отдельного элемента, дублирует контент. Отклонено.

### 2. Двухшаговая замена DOM в `flushHtmlToDOM`

**Решение**: `element.innerHTML = ''` → `element.innerHTML = html`.

**Почему**: `aria-live` реагирует на добавление узлов в DOM (мутации типа `childList`). Прямая замена `innerHTML` одним присваиванием в некоторых браузерах/скринридерах воспринимается как «удаление и добавление», но порой недостаточно надёжно. Двухшаговый подход гарантирует явный `childList` мутацию на добавление.

**Почему не `requestAnimationFrame`**: добавляет сложность и задержку — не нужно, если двухшаговый подход работает синхронно.

### 3. Фокус после рендера — на первую кнопку, не на `#scene-content`

**Решение**: `focusAfterRender` больше не вызывает `sceneEl.focus()` — он всегда ищет первую кнопку (act → goto). `#scene-content` читается через `aria-live`, а не через фокус.

**Устраняет двойное зачитывание**: именно этого опасался старый спек. Теперь `aria-live` объявляет контент сцены, а фокус идёт на кнопку — двойного чтения нет.

**Кейс без кнопок**: если сцена не содержит ни act, ни goto кнопок (нестандартно, но возможно) — фокус остаётся там, где был. `#scene-content` в этом случае можно оставить как запасной вариант с `tabindex="-1"`.

## Risks / Trade-offs

- **`aria-live` + `aria-atomic` читает весь текст сцены целиком** — при длинных сценах это может быть долго. Митигация: это ожидаемое поведение IF-игры; игрок всегда может прервать скринридер.
- **Поведение без кнопок** — если нет ни act, ни goto, фокус «теряется». Нестандартный кейс, заложим запасной `sceneEl.focus()` как последний fallback.
- **Изменение поведения для зрячих клавиатурных пользователей** — фокус больше не прыгает на контейнер сцены, сразу на кнопки. Для IF это лучше: пользователь сразу готов к действию.
