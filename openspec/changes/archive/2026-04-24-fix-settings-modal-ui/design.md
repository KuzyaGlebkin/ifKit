## Context

В `Settings` сейчас полей для акцентного цвета **нет** — оформление использует CSS-токен `--ifk-color-accent` из `style.css` и (по `ui-theming`) автор/игрок могут перезаписывать переменные, но **персистентного** выбора в модалке нет. Модалка (`settings-modal.ts`) рядом с ползунками выводит «сырые» числа (`0.8`, `1.0×` как множитель, доли 0–1), что пользователи ожидают видеть **в процентах**. Для `language: ''` (авто) ни одна кнопка не получает `ifk-lang-btn--active`, т.к. в группе только конкретные коды; при этом `initI18n(..., '')` выбирает рабочий язык, так что **сохранённая** настройка `''` и **активный** `_lang` в рассинхроне. Секция «Язык» построена на `fieldset` + `div.ifk-lang-group` внутри `.ifk-settings-row` — на узких/многоязычных вариантах сегменты могут переноситься не так, как соседние строки (Тема / ползунки с фиксированной подписью 5rem).

## Goals / Non-Goals

**Goals:**

- **Акцент:** добавить в настройки игрока **минимальное, но полное** управление: пресет акцентного цвета, сохраняемый в `Settings` и применяемый в `applySettings`, плюс группа в модалке, визуально согласованная с «Тема».
- **Ползунки:** хранимые и поведенческие значения оставить **как в типе** (`0.8–1.4` для шрифта, `0–1` для громкостей), менять **только подписи** рядом с ползунком на **проценты** (для громкости: 0–100%; для шрифта: 80%–140% с **100% = 1.0** множителю, без дробей в духе «1.0×»).
- **Язык:** одна **явная** кнопка «Авто» (или тот же смысл в i18n), `data-lang-value=""`, визуально выделенная, когда `settings.language === ''`; при смене на `''` заново применяется та же **логика, что в `initI18n` для пустой строки**; вёрстка: группы кнопок языка на одной линии с подписью «Язык» в типичной ширине окна.
- **Спеки:** дельта к `settings-modal`, `user-settings`, `ui-theming` (только уточнения по применению акцента), `i18n` (контракт смены авто-языка).

**Non-Goals:**

- Собственный color picker / произвольный hex (только пресеты).
- Менять **нижележащие** диапазоны/шаг `input[type=range]` (минимальные дифы — в подписях; при желании `aria-valuetext` в процентах, без смены модели `Settings`).

## Decisions

1. **Акцент — пресет в `Settings`**

   - **Что:** новое поле, например `accent: 'default' | 'blue' | 'violet' | 'emerald'` (точные имена и палитра — в реализации; дефолт `default` = **не** инлайн-переопределять токены, чтобы работали значения из `style.css` и тёмные переопределения).
   - **Почему:** соответствует `ui-theming` (единый токен) и вопросу «функционал» — **ранее не хранилось** в `Settings`, только CSS; теперь **игровой** слайс через `applySettings` + дублирование `engineDefaults` / `mergeDefaults` / JSON экспорт-импорт.
   - **Альтернативы:** (а) не добавлять UI и только документировать — **отклонена**, т.к. в запросе ожидается переключатель; (б) свободный `setProperty` из поля hex — **отклонена** (сложность и безопасность/контраст).

2. **Подписи ползунков в процентах**

   - **Музыка/звук:** `Math.round(volume * 100) + ' %'`.
   - **Шрифт:** `Math.round(fontSize * 100) + ' %'` (80% … 140%), единообразно с «сотнями» для громкостей; при необходимости уточнить в `aria-valuetext` тот же текст.
   - **Почему:** внутренняя модель не ломается, меняется только **отображение**.

3. **Авто-язык: кнопка + согласованность с i18n**

   - **Что:** в списке языков (при 2+ локалях) первая кнопка «Авто» с `data-lang-value=""`, активна при `settings.language === ''`.
   - **Поведение при выборе «Авто»:** после сохранения настроек **перезапустить логику** выбора языка как `initI18n(locales, '')`. Чтобы не дублировать, лучше: экспорт из `i18n` одной функции, например `setLanguageOrAuto('' | 'ru' | 'en'…)`, внутри для `''` вызвать ту же ветку, что `initI18n` (возможен рефакторинг: общая `resolveAndApplyLanguage` с доступом к сохранённым `locales` в модуле).
   - **Почему:** `setLanguage('')` **один** в текущем `i18n` ломает `t` (см. текущую реализацию) — **исправление** входит в объём; без этого «Авто» непригоден.
   - **Альтернатива:** маппить `''` на **резолв** и хранить только конкретный код — **отклонена** (ломает семантику «следовать браузеру/первому коду» при смене окружения и контрастирует с `engineDefaults.language: ''`).

4. **Вёрстка «Язык»**

   - Держать `.ifk-lang-group` в `flex` с `flex-wrap: nowrap` (и при **очень** узком диалоге — осторожный `overflow-x: auto` *или* допустимый перенос, но **не** отличающийся от других сегментов без причины); убедиться, что `fieldset` выравнивается с `.ifk-settings-row` (подпись 5rem + `flex: 1; min-width: 0` у группы).

5. **Тема/скринридер**

   - Существующие `aria-hidden` на блоке с темой/шрифтом (по `openspec/specs/settings-modal`) **не** ослабляем в рамках этого change (акцент по умолчанию — **видимая** сегмент-группа уровня «Тема»; уточнение в сценариях, если копируем модель ARIA с теми же правилами).

## Risks / Trade-offs

- [Расширение `Settings` ломает старые JSON без `accent`] → `mergeDefaults` + дефолт `default` при `load` / merge.
- [Слишком много кнопок в языковой группе (Авто + N)] → визуально плотнее или горизонтальный скролл на xs.
- [Дублирование палитры акцентов в `applySettings` и в CSS] → единая таблица констант в одном TS-модуле, применяемом из `applySettings` только.
- [Регресс `setLanguage`] → покрыть смену `en` ↔ `''` ↔ `ru` и `rerender`.

## Migration Plan

- **Deploy:** релиз как обычно; в хранилище существуют магазины без `accent` — **merge** с `engineDefaults`.
- **Rollback:** откат коммита; несовместимость с новым полем не ожидается при откате (лишнее поле в JSON игнорируется старым кодом, если кто-то останется на старой ветке — наоборот, теряет accent, приемлемо).

## Open Questions

- **Лейбл «Авто»:** дословно Russian в шаблоне сейчас, или через `t` после того, как i18n стабилен (может вынести в константу/ключ позже).
- **Точная палитра** пресетов акцент (hex) — выбрать **контраст** к `btn-goto` / WCAG, под механизм `ui-theming` (отдельные `accent-fg` на тёмных оттенках в таблице при необходимости).
