## Context

Этап 0 создал скаффолдинг: структуру каталогов, конфигурацию Vite, шаблоны `game.ts` / `scenes.ts` / `state.ts` и заглушку `defineGame()`. Сейчас `defineGame()` ничего не делает. Нет буферов, нет теговых функций, нет перехода между сценами, нет кнопок действий.

Этот документ фиксирует технические решения для модулей: `renderer.ts`, `tag-functions.ts`, `scenes.ts`, `engine.ts`.

> Решения 1–4 — из первоначального проектирования. Решения 5–7 добавлены по итогам анализа в `docs/analysis/etap-1-rendering.md`.

## Goals / Non-Goals

**Goals:**

- Реализовать три буфера рендеринга и механизм их сброса в DOM
- Реализовать теговые функции двух видов (императивные и функциональные)
- Реализовать тип `Scenes<S>`, игровой цикл, функции-билдеры `act` и `goto`
- Связать `defineGame()` с новым движком

**Non-Goals:**

- Финальные UI-зоны и разметка — Этап 3
- Управление состоянием (`GameState`) — Этап 2
- Сохранения, звук, i18n, настройки — более поздние этапы
- Асинхронные сцены и анимации переходов

## Decisions

### 1. Формат буферов: строки и массивы объектов

**Решение**: три модульных буфера:
- `htmlBuffer: string` — накапливает HTML от теговых функций
- `actBuffer: ActionDef[]` — накапливает кнопки действий от `act()`
- `gotoBuffer: ActionDef[]` — накапливает кнопки переходов от `goto()`

где `ActionDef = { label: string; handler: () => void }`.

**Обоснование**: `htmlBuffer` применяется через `innerHTML` — просто и без зависимостей. `actBuffer` и `gotoBuffer` — массивы объектов, потому что при сбросе в DOM нужно создать `<button>` с обработчиком click. Обработчики (`handler`) — заранее замкнутые функции: они уже знают о текущем `state` и о том, что делать по клику.

### 2. Буферы — модульные переменные, не контекстный объект

**Решение**: все три буфера живут в модуле `renderer.ts`. Теговые функции и функции-билдеры импортируются напрямую.

**Обоснование**: JavaScript однопоточный. Передавать контекст в каждый вызов (`P(ctx, 'Текст')`) — лишний cognitive overhead. Автор пишет `P('Текст')`, `act('Взять меч', ...)`, `goto('clearing', 'На поляну')` без ссылки на контекст.

### 3. Две формы теговых функций

**Решение**: `P('текст')` пишет в `htmlBuffer`; `p('текст')` возвращает строку `'<p>текст</p>'`.

**Обоснование**: императивная форма (`P`) — основная для тела сцены. Функциональная форма (`p`) нужна для композиции: `P(strong('Жирный') + ' обычный')`.

### 4. Type-safe ключи сцен через generic контекст

**Решение**: сцены получают второй аргумент `ctx: SceneContext<S, K>`, где `K = keyof typeof scenes`. TypeScript инферит `K` из объекта сцен в `defineGame()`.

**Обоснование**: `ctx.goto('nonexistant', ...)` выдаёт ошибку компиляции. Автодополнение показывает допустимые ключи. Подход не требует отдельной `registerScenes()` — `defineGame()` сам создаёт контекст. `goto` не нужно импортировать отдельно.

**Альтернатива**: глобальная переменная `goto`, переопределяемая при инициализации — проще в реализации, но без type-safety и автодополнения. Отклонено.

### 5. `act` и `goto` — функции-билдеры

**Решение**:
- `act(label, cb)` — создаёт кнопку в `actBuffer`. По клику: `cb(state)` → перерисовать текущую сцену.
- `goto(key, label, cb?)` — создаёт кнопку в `gotoBuffer`. По клику: `cb?(state)` → перейти к сцене `key`.

**Обоснование**: в текстовой адвенчуре нельзя сразу переходить к сцене при вызове `goto` — нужно дождаться выбора пользователя. Оба вызова декларируют намерения во время рендера, движок рендерит кнопки, пользователь кликает — происходит действие. Одинаковая сигнатура `act`/`goto` снижает когнитивную нагрузку.

### 6. `static` получает тот же контекст `{ act, goto }`

**Решение**: `static` в `defineGame()` имеет сигнатуру `(state: S, ctx: SceneContext<S, K>) => void` — идентично сценам.

**Обоснование**: `goto` в `static` нужен для глобальных переходов (например, использование свитка телепортации из инвентаря). `act` нужен для глобальных действий (выпить зелье из любой сцены). Кнопки из `static` попадают в те же `actBuffer`/`gotoBuffer` и рендерятся в зонах сцены — визуально доступны у нижней части экрана.

**Порядок**: `static` вызывается первым → его кнопки в буфере первыми → в Этапе 3 UI-порядок будет скорректирован.

### 7. Первая сцена — первый ключ объекта `scenes`

**Решение**: при старте вызывается `runGameLoop(Object.keys(scenes)[0])`.

**Обоснование**: явный `startScene` — лишнее поле для большинства игр. Автор контролирует порядок ключей в объекте. При необходимости можно добавить `startScene` в Этапе 2+.

## Risks / Trade-offs

- **Строковый буфер уязвим к XSS** → В v1 доверяем автору: игра — самодостаточный HTML-файл.
- **Глобальные буферы усложняют тестирование** → Экспортируем `clearBuffers()` и `getHtmlBuffer()` для сброса и инспекции в тестах.
- **`Object.keys(scenes)[0]` зависит от порядка ключей** → В современном JS порядок строковых ключей детерминирован (порядок вставки). Приемлемо для v1.
- **`act` перерисовывает текущую сцену полностью** → Предсказуемо и просто. Частичные обновления — вне скоупа v1.
