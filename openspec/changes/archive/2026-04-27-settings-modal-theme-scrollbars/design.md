## Context

Модалка настроек собрана из `#ifk-settings-dialog` с фиксированной шапкой (`.ifk-modal-header` + `#ifk-settings-close`) и блока `.ifk-modal-body` со всем контентом. Сейчас `.ifk-modal-dialog` задаёт `overflow: hidden` без ограничения высоты относительно viewport: при росте контента (в т.ч. из‑за ползунка шрифта выше 100%) диалог может стать выше экрана, и кнопка ✕ уезжает за пределы видимой области. Модалка сохранений уже использует `.ifk-saves-body { overflow-y: auto; max-height: 60vh }` — для настроек нужен согласованный, но более надёжный предел по высоте (желательно `dvh`/`vh` с учётом padding backdrop).

Полосы прокрутки: в Firefox срабатывает `scrollbar-color`; в Chromium до недавних версий основной стилизацией остаются псевдоэлементы `::-webkit-scrollbar` (и при необходимости дублирование через `scrollbar-color` для единообразия). Токены `--ifk-color-surface`, `--ifk-color-border`, `--ifk-color-text` достаточны для трека и «бегунка»; при желании вводятся отдельные `--ifk-scrollbar-track` / `--ifk-scrollbar-thumb`, привязанные к этим токенам в `:root` и `[data-theme="dark"]`, чтобы не дублировать бизнес-логику темы.

## Goals / Non-Goals

**Goals:**

- Полосы прокрутки в интерфейсе движка (как минимум в прокручиваемых контейнерах модалок и глобально для `html`/`body` при необходимости) визуально согласованы со светлой/тёмной темой в Chromium и Firefox.
- Модалка настроек не превышает доступную высоту окна: контент под шапкой прокручивается; кнопка закрытия остаётся в шапке и доступна указателю без обходного прокручивания всего окна браузера.

**Non-Goals:**

- Пиксельная идентичность скроллбара во всех браузерах и ОС (нативные overlay-скроллбары macOS могут вести себя иначе).
- Отдельная настройка «толщины» скроллбара пользователем.
- Изменение логики фокус-ловушки модалки, кроме косвенного эффекта от новой прокручиваемой области.

## Decisions

1. **Ограничение высоты диалога настроек**  
   Задать для `.ifk-modal-dialog` максимальную высоту относительно viewport, например `max-height: min(100dvh - 2 * padding_backdrop, …)` с fallback на `100vh` где `dvh` не нужен. Включить flex-колонку: тело с `flex: 1 1 auto`, `min-height: 0`, `overflow-y: auto` — стандартный паттерн для скролла внутри flex-ребёнка.

2. **Глобальные токены скроллбара**  
   Добавить в `style.css` на `:root` / тёмную тему переменные `--ifk-scrollbar-thumb` и `--ifk-scrollbar-track` (или один токен трека + использование border/surface), применить:
   - `scrollbar-width: thin` (уже есть в части правил — унифицировать);
   - `scrollbar-color: <thumb> <track>` на `html` или универсальном селекторе;
   - для WebKit — `::-webkit-scrollbar` / `::-webkit-scrollbar-thumb` / `::-webkit-scrollbar-track` с теми же переменными.

3. **Область применения webkit-правил**  
   Ограничить глобальные webkit-правила минимально достаточной областью (`html`, или `html, .ifk-modal-body, .ifk-saves-body`), чтобы не переопределять лишние внутренние контейнеры без необходимости; при этом страница игры и модалки должны выглядеть согласованно.

**Альтернативы:**

- Только `scrollbar-color` без webkit — отклонено для исходной проблемы Chrome.
- Только локальные стили на `.ifk-modal-body` без глобальной темы — недостаточно для «белого» скролла вне модалок.

## Risks / Trade-offs

- **[Risk] `100dvh` и старые браузеры** → Указать fallback `100vh` рядом или через `@supports`.
- **[Risk] Слишком тёмный/светлый thumb на границе контраста** → Брать thumb от `--ifk-color-border` / surface mix, при необходимости слегка отличать от чистого border; проверить в обеих темах.
- **[Trade-off] Глобальный webkit-scrollbar** может затронуть нежелательные элементы** → Сужение селекторов или наследование только от корня.

## Migration Plan

Изменения только в статических ассетах движка (`style.css`, при необходимости разметка модалки). Откат — revert коммита. Миграции данных нет.
