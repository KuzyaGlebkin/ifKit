## Context

Карточка занятого именного слота в `saves-modal.ts` использует `role="group"` и `aria-labelledby`, указывающий только на `ifk-slot-num` и при наличии — на `ifk-slot-label`. Строки превью act/goto выводятся в `.ifk-slot-info` как `.ifk-slot-preview` без собственных `id` и не участвуют в `aria-labelledby`, поэтому при типичном обходе (имя группы → следующий фокусируемый узел) вспомогательные технологии не сшивают эти строки с «именем» карточки так же, как номер и сцену, что воспринимается как пропуск.

## Goals / Non-Goals

**Goals:**

- Включить непустые превью `actPreview` и `gotoPreview` в тот же механизм, что задаёт доступное имя группы (или эквивалент с тем же эффектом для пользователя: строки не теряются перед кнопками).
- Сохранить видимую вёрстку и классы, используемые стилями; изменения — минимальные, в одном месте рендера слота.
- Дать клавиатурным пользователям перемещение **↑/↓** по строкам информации и кнопкам внутри одной карточки без раздувания порядка **Tab**.

**Non-Goals:**

- Менять формат хранения `SaveSlot` или правила снятия превью в `saves.ts`.
- Подсказки/лейаут модалок «Настройки» и scrollbar (другой запрос).
- Добавлять строки `.ifk-slot-info` в обычный порядок **Tab** (по-прежнему ориентир — кнопки; строки достижимы фокусом со `tabindex="-1"` через стрелки с соседних участников цепочки).

## Decisions

1. **Расширить `aria-labelledby` на id строк превью**  
   Для каждого непустого превью act/goto ренерить `span` с **стабильным** `id` (например `ifk-slot-act-{index}` / `ifk-slot-goto-{index}`) и добавлять эти id в тот же атрибут `aria-labelledby` **после** id номера и, если есть, id подписи сцены, в соответствии с порядком данных в UI (act перед goto, как сейчас в разметке).

2. **Почему не только `aria-describedby`**  
   Имя группы, собранное из `aria-labelledby`, последовательно зачитывается как связное имя карточки; отдельный `describedby` ведёт себя иначе в комбинациях с NVDA/JAWS. Расширение списка id в `aria-labelledby` согласуется с уже принятой в спеке моделью «group + labelledby» и с минимальным диффом.

3. **Почему id на каждой строке, а не один скрытый текст**  
   Соответствие видимому тексту дублированием в off-screen блоке рискует рассинхронизацией; ссылки на те же узлы, что видит зрячий пользователь, дешевле в сопровождении.

4. **Вертикальная навигация ArrowUp / ArrowDown**  
   Цепочка фокуса: прямые дети `.ifk-slot-info` в DOM-порядке, затем прямые дети `.ifk-slot-actions` (так согласованы «строки сверху вниз» и кнопки столбцом). Обработчик `keydown` на корне `.ifk-slot` (capture не обязателен), если `event.target` — элемент цепочки (длина ≥ 2) и клавиша ArrowUp/ArrowDown: `preventDefault`, фокус на соседний элемент по кольцу. Не-кнопки получают `tabindex="-1"` при привязке к карточке и повторно после подмены HTML в `showDeleteConfirm` / `showOverwriteConfirm`. Автослот использует тот же механизм (только свои строки + кнопка «Загрузить» при наличии).

5. **Почему стрелки не вносят строки в Tab**  
   Иначе N слотов × несколько строк дали бы длинный Tab-тракт; сценарий — Tab по модалке как раньше, чтение строк клавиатурой — через **↑/↓** с кнопки (например **↑** с «Загрузить» ведёт к последней видимой строке в info).

## Risks / Trade-offs

- **[Risk]** Слишком длинное склеенное имя группы при длинных превью → **Mitigation:** то же усечение, что уже в данных (80 символов) и визуальный ellipsis; при необходимости в будущем — сократить объявляемую строку отдельной задачей.
- **[Risk]** Скринридер при поэлементном обходе строк снова озвучит текст, уже входящий в имя `aria-labelledby` → **Mitigation:** приёмлемо (типично для `tabindex="-1"` на сочетаемом с group label); при дублировании в конкретном AT — уточнить отдельно.
