## Context

ifKit рендерит сцены через `flushHtmlToDOM(sceneEl)` — весь HTML зоны описания заменяется целиком при каждом рендере. Элементы зоны описания — произвольные блочные теги (`<p>`, `<h1>`–`<h3>`, `<ul>`, `<img>`, `<hr>` и т.д.). Хранилище движка — `localStorage`; уже используются ключи `ifkit:settings` и `ifkit:saves`. Система настроек полностью сформирована.

## Goals / Non-Goals

**Goals:**
- Блок подсвечивается в момент первого отображения; повторный показ того же контента — без подсветки
- Хранилище виденного персистентно (localStorage) и глобально (не зависит от слотов сохранений)
- Пользователь может отключить подсветку в настройках
- Автор может задать `settings: { showUnseenHighlight: false }` как умолчание

**Non-Goals:**
- Подсветка на уровне отдельных слов или предложений
- Аналитика (счётчик непрочитанных)
- Экспорт/импорт хранилища seen вместе с основным экспортом сохранений

## Decisions

### D1: Хранилище виденного — отдельный ключ localStorage, вне snapshot

**Выбрано**: `ifkit:seen` — отдельный ключ, не включается в `Snapshot` (history/saves).  
**Альтернатива**: часть состояния игры / snapshot.

Rationale: семантика «виденного» глобальна и трансигровая — как в Ren'Py. Загрузка старого сохранения не должна «забывать» что игрок уже читал. Undo/redo не должны восстанавливать подсветку.

### D2: Хешируем innerHTML блока, не outerHTML

**Выбрано**: `el.innerHTML` как строка для хеширования.  
**Альтернатива**: `el.outerHTML`.

Rationale: `outerHTML` включает класс `paragraph--unseen` который мы сами устанавливаем — хэш менялся бы от рендера к рендеру. `innerHTML` стабилен относительно наших CSS-классов.

### D3: Простой целочисленный хеш (djb2-подобный)

Используем `Math.imul(31, h) + charCodeAt(i)` с финальным `.toString(36)`. Не криптографический — нужна только скорость и компактность (~6–8 символов).

### D4: Хранилище — `Record<sceneKey, string[]>` (массив хешей)

Формат: `{ "room_forest": ["a1b2c3", "d4e5f6"], ... }`. Массив (не Set) — JSON-сериализуем напрямую. Дубликаты невозможны: один и тот же хеш одного и того же sceneKey встречается только при идентичном тексте, значит блок уже был виден — добавлять повторно смысла нет.

### D5: markAndHighlight вызывается после flushHtmlToDOM, до focusAfterRender

Место в `runGameLoop` в `scenes.ts`:
```
flushHtmlToDOM(_sceneEl)       // устанавливает innerHTML
markAndHighlight(_sceneEl, currentSceneKey)  // ← здесь
flushActsToDOM(_actsEl)
flushGotosToDOM(_gotosEl)
focusAfterRender(...)
```

Это гарантирует что классы установлены до передачи фокуса.

### D6: CSS — fade-out анимация через CSS-переменную

```css
.paragraph--unseen {
  background-color: var(--ifk-unseen-color, rgba(255, 220, 80, 0.18));
  border-radius: 3px;
}
```

Без анимации в базовой версии — достаточно статичного фона. Автор может переопределить `--ifk-unseen-color` в своём CSS.

### D7: showUnseenHighlight проверяется в markAndHighlight

Функция читает текущие настройки (через `loadSettings`) или получает флаг аргументом. Предпочтительно передавать флаг аргументом — нет скрытой зависимости от хранилища.

## Risks / Trade-offs

**[Risk] Рост хранилища**: bounded размером игры (фиксированное число параграфов). Не является реальной проблемой.

**[Risk] Смена языка вызывает ложную подсветку**: `innerHTML` изменился из-за перевода, не из-за новой ветки.  
→ Mitigation: это корректное поведение — переведённый текст технически новый. Дополнительной обработки не требуется.

**[Risk] `markAndHighlight` вызывается при undo/redo**: исторические состояния рендерятся заново.  
→ Mitigation: блоки помечаются как виденные при первом рендере; undo/redo показывает тот же текст снова — он уже в хранилище seen, подсветки не будет. Поведение корректное.
