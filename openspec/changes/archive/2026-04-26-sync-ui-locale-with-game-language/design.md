## Context

Сейчас `locales` в `defineGame` задают только **игровой** словарь для `t`. Подписи движка (модалки, панель `#controls`, inline-подтверждения) захардкожены на одном языке, а `index.html` фиксирует `lang="ru"`. Настройка `language` обновляет `t`, но не меняет объявляемую локаль документа и не трогает UI Chrome.

## Goals / Non-Goals

**Goals:**

- Ввести **системный (UI) словарь** в составе предоставляемых автором локалей: стабильные **ключи** (например `settings.title`, `toolbar.undo`) → строка на каждом языке.
- **Единый resolved language** (как для `t`, включая `''` → авто) определяет: строки UI, атрибут `lang` на `<html>`, при необходимости `title` / `aria-label` движка.
- **Слияние с встроенными** строками движка: если автор не передал `ui` для языка или отдельного ключа, использовать **встроенный** набор (как минимум для языков, которые движок официально поддерживает в шаблоне), чтобы пустой `ui` **не** ломал интерфейс.
- При смене языка **перерисовка/обновление** DOM-узлов, где отображается движковый текст (модалки, тулбар), без перезагрузки страницы.

**Non-Goals:**

- Перевод **контента сцен** вне `t` (уже покрыто).
- Локализация **сообщений об ошибках разработчика** (`throw new Error` в dev) и внутренних `console` — остаются на языке кода/русской диагностике, если не решено иначе позже.
- **Snackbar**-форматтеры из `watch` — остаются ответственностью автора (строки игры); отдельный движковый pass не вводим в этом change, если не используют `t`/`u`.

## Decisions

1. **Форма данных в `defineGame`**

   - **Предпочтительно**: каждый язык в `locales` — объект `{ game: Record<string, string>, ui: Record<string, string> }`, где `game` — прежние пары для `t`, `ui` — системные ключи.
   - **Обратная совместимость**: если по языку передан **плоский** `Record<string, string>`, движок SHALL трактовать его как **только `game`**, а `ui` взять из **merge( built-in defaults[lang], … )** по ключам. Так старые игры без `ui` продолжают работать, но интерфейс остаётся на встроенных переводах до тех пор, пока автор не дополнит `ui`.

2. **API поиска строк UI**

   - Ввести функцию уровня движка, например `u(key: string): string` (имя согласовать с код-стайлом проекта), которая: берёт `ui` для **активного resolved** языка; при отсутствии ключа — fallback на **встроенный** `ui` для того же языка, затем на primary subtag, затем на разумный дефолт (например en); последний — сам ключ или русская/английская встроенная константа (зафиксировать в реализации).

3. **Связь с `sourceLanguage` и `fullLocales`**

   - Для `sourceLanguage` пустой слой `{}` в `game` остаётся; для `ui` SHALL подмешиваться встроенный слой для **этого** кода языка, чтобы UI не пустел.

4. **`<html lang>`**

   - Значение SHALL быть **BCP 47 primary language subtag** (например `en`, `ru`), совпадающим с логикой «активного языка» i18n (тот же идентификатор, что и для выбранной локали `game`/`t`).
   - Обновлять: при `initI18n`, при `setLanguage` / `setLanguageOrAuto`, при `applySettings` после загрузки настроек, чтобы импорт/старт согласовали `lang` без дрейфа.
   - **Начальный** `index.html`: допустимы `lang="en"` (нейтральный) или `und`; в любом случае движок **перезаписывает** при первом `applySettings`/`initI18n`. Предпочтение: **не** оставлять вечно `ru`, если игра на `en`.

5. **Где обновлять подписи**

   - `settings-modal.ts`, `saves-modal.ts`, инициализация/обновление текстов **кнопок `#controls`** (через существующие id или выделенную функцию `syncChromeI18n()`), заголовки модалок, подписи `lang`-секции — всё через `u(...)` (или data-атрибуты + одна перерисовка).
   - Кнопки с **кодом языка** (`ru`, `en`) остаются **лейблами-кодами**, а «Авто» — через `u('lang.auto')`.

6. **Список ключей**

   - Централизовать константы ключей (например `UI_KEYS` в одном файле) и перечислить в `tasks.md` / спеках, чтобы авторы копировали в свои `ui`-словари. Минимум: всё, что сейчас на русском в `settings-modal`, `saves-modal`, `index.html` для toolbar.

**Alternatives considered**

- *Только `t` с префиксом ключей `__ui.*`* — смешивает пространства имён с игровыми строками; отклонено в пользу отдельного `ui` и `u()`.
- *Отдельный файл `ui.json` вне `defineGame`* — хуже для портативной игры и одного артефакта; остаётся вариантом для шаблона, но не обязателен в первом шаге.

## Risks / Trade-offs

- **[Risk]** Авторы забывают заполнить `ui` — **mitigation**: встроенные defaults + документированные ключи.
- **[Risk]** Расхождение `lang` и фактического текста при частичных переводах — **mitigation**: приоритетный fallback `u` и тесты на смену языка.
- **[Trade-off]** Плоский старый `locales` не даёт UI-переводов от автора — только встроенные; приемлемо для backward compatibility.

## Migration Plan

- Существующие проекты без `ui` продолжают запуск; интерфейс на **встроенных** словарях движка.
- Авторы, которым нужен полный контроль, постепенно добавляют `ui` per language.
- Rollback: откат коммита; риск низкий (feature-слой).

## Open Questions

- Точный список **встроенных** языков движка (минимум `ru`, `en`?) — зафиксировать при реализации по текущему `navigator` / целевой аудитории.
- Нужен ли `document.querySelector('html')` vs `document.documentElement` — только стиль кода, без спек-различия.
