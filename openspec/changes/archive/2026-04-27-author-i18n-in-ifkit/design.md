## Context

Сейчас `t` из `src/ifKit/i18n.ts` — tagged template, строящий ключ для слоя `game` в `locales`. Разметка (`P`, `H1`, `em`, `strong` в `tag-functions.ts`) принимает **уже готовую** строку. Авторская сцена сочетает вызовы: `P(t\`…\`)`, вложенно `em(t\`…\`)`. Образец в `src/scenes.ts` импортирует `t` наряду с тегами.

## Goals / Non-Goals

**Goals:**

- Дать в ifKit **один шаг** от исходной фразы до вставки в буфер (или до inline-HTML), с **тем же** поведением перевода, что у `t` (ключи, плейсхолдеры `{0}`…, fallback).
- Экспортировать helpers из публичного `index.ts`, чтобы типичная `scenes.ts` **не** импортировала `t` для сценовой прозы и заголовков.
- **BREAKING:** снять с публичного API прежние «сырые» примитивы разметки (приём готовой `string` под именами `P`/`H1`/`em`/`strong` и т.д.). Редкие обходы — только внутри движка или через явно другой, нестабильный контракт, не в форме старого API.

**Non-Goals:**

- Менять формат `locales` в `defineGame` или логику `initI18n` / `setLanguageOrAuto`.
- Заменять `u()` для Chrome/UI.
- В этой итерации **не** требовать оборачивать подписи `act` / `goto` в новый API (можно оставить `act(t\`…\`, cb)`; см. Open Questions).
- Менять контракт `SceneContext.act` / `goto` на tagged-template без отдельного анализа типов.

## Decisions

### 1. Имена API: один публичный контракт «tagged + `t`»; старое string-API убрать

**Выбор (один из эквивалентов, фиксируется при реализации):**

- **Вариант A:** `Pt`, `H1t`, `H2t`, `H3t` + `emt`, `strongt`; публичные `P`/`H1`/`em`/`strong` с `(content: string)` **удалить** с экспорта.
- **Вариант B:** `P`/`H1`/… **переопределить** на **только** tagged template (ломает `P("…")` и `P(t\`…\`)` в прежнем виде — миграция на `` P`…` ``).

**Почему не дублировать `P` и `Pt`:** **BREAKING** принят намеренно; два параллельных публичных пути не требуются.

### 2. Реализация: общая внутренняя функция «tagged `t`»

**Выбор:** внутри ifKit (напрямую или через тонкую обёртку) вызывать **тот же** `t(strings, ...values)` из `i18n.ts`, затем передавать результат в **внутренние** `p`/`h1`/`em`/`strong` (необязательно публичные) и `appendHtml` / возврат строки.

**Почему:** дублировать алгоритм ключа — источник расхождений с `t`.

### 3. Образцовая игра

**Выбор:** перевести `src/scenes.ts` на `Pt`/`H1t`/… и `emt`/`strongt` где уместно; в `src/game.ts` оставить `t` для `watch` и прочей логики вне сценовой разметки (или эквивалент через helpers, если появятся — не обязательно в v1).

## Risks / Trade-offs

- **[Риск] Неполный набор тегов** — нет `At` / `LIt` в первой версии; автору может понадобиться `t` и/или внутренние примитивы, пока не расширим публичный набор. **Смягчение:** v1 = заголовки H1–H3, P/em/strong (как tagged); follow-up.
- **[Трейд-офф]** `act`/`goto` с метками: без расширения контекста остаётся `t` вручную — согласовано с narrow scope.
- **[BREAKING]** Любой внешний копипаст/форк, использовавший `P`/`H1`/`em`/`strong` со `string` или `` P(t`…`) ``, **сломается** до миграции; это приемлемо и зафиксировано.

## Migration Plan

1. Реализовать tagged API и **снять** публичные старые примитивы (или заменить сигнатуры несовместимо).
2. Обновить образец `scenes.ts` / `game.ts` и весь внутренний код движка, ссылающийся на старые вызовы.
3. Задокументировать миграцию: «было `` P(t`…`) ``, стало `` P`…` `` / `Pt\`…\`` (в зависимости от выбранного варианта имён)».

**Откат:** только откат коммита/релиза; **не** обещаем длить параллельно старый и новый публичный API.

## Open Questions

- Нужны ли в v1 also imperative `Em`/`Strong` с переводом (`Emt`/`Strongt` vs только lowercase `emt` для композиции), чтобы совпасть с двойным API в `tag-functions.ts`.
- Стоит ли в отдельном change рассмотреть `act` / `goto` с tagged label (требует дизайна API на `SceneContext`).
