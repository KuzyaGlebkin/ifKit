# Решения: Этап 2 — Состояние игры

## Глобальное состояние

Автор передаёт объект начального состояния в `defineGame()`. Движок хранит ссылку на него и передаёт в каждую сцену первым аргументом. Мутации происходят напрямую — без обёрток, прокси и геттеров.

```ts
// state.ts (пользовательский файл)
export const initialState = {
  hp: 10,
  gold: 0,
  hasKey: false,
}
export type GameState = typeof initialState
```

```ts
// scenes.ts
export const scenes = {
  cave: (state: GameState, ctx) => {
    P(`HP: ${state.hp}`)
    ctx.act('Выпить зелье', (s) => { s.hp += 5 })
  },
}
```

Тип `GameState` выводится из `typeof initialState` — автор не объявляет тип вручную, он следует из данных.

## Локальное состояние сцены

Некоторые переменные нужны только пока игрок находится в одной сцене и должны сбрасываться при переходе в другую. Пример: счётчик повторных осмотров комнаты.

Проблема простых `let` в теле сцены: при каждом ре-рендере (после `act`) функция сцены вызывается заново — переменная сбрасывается.

**Решение:** `ctx.local(defaults)` — движок хранит объект, привязанный к ключу текущей сцены, и возвращает его при каждом вызове.

```ts
room: (state, ctx) => {
  const local = ctx.local({ searched: 0 })

  P(`Ты в комнате. Обысков: ${local.searched}`)

  ctx.act('Обыскать', () => { local.searched++ })
},
```

- `ctx.local(defaults)` принимает объект с начальными значениями; тип выводится автоматически
- При первом входе в сцену создаётся новый объект из `defaults`
- При ре-рендере той же сцены возвращается существующий объект
- При `goto` в другую сцену объект текущей сцены удаляется
- При откате (undo) объект восстанавливается из снимка истории вместе с `state`

Данные `sceneLocals` включаются в снимок истории наравне с глобальным состоянием.

## Клонирование состояния

`JSON.parse(JSON.stringify(...))` — везде: и для снимков истории, и для слотов сохранений.

Единственный механизм — единственное ограничение — нет асимметрии между «работает в памяти» и «ломается при сохранении».

**Ограничение на типы данных в `GameState` и `sceneLocals`:**

> Состояние должно быть JSON-совместимым деревом данных: примитивы, массивы, вложенные объекты. Запрещены: `Date`, `Map`, `Set`, `undefined`, функции, классовые экземпляры, циклические ссылки.

Для IF-игры это естественное ограничение: состояние — данные, а не поведение.

Отклонённые варианты:

| Вариант | Причина отклонения |
|---|---|
| `structuredClone` для истории, `JSON` для сохранений | Асимметрия: `Set` работает в истории, но молча ломается при сохранении |
| `structuredClone` везде | `localStorage` требует строку — `JSON` неизбежен для сохранений; проще унифицировать |

## Snackbar — уведомления об изменении характеристик

Автор объявляет отслеживаемые ключи и функции форматирования в `defineGame()`:

```ts
defineGame({
  state: initialState,
  watch: {
    hp:   (prev, next) => `HP: ${prev} → ${next}`,
    gold: (prev, next) => next > prev ? `+${next - prev} золота` : `-${prev - next} золота`,
  },
  scenes,
})
```

- Если функция вернула `null` или `undefined` — уведомление не показывается
- Движок сам делает дифф: запоминает значения отслеживаемых ключей до колбэка `act`/`goto`, сравнивает после
- Дифф только по плоским примитивам (`===`) — вложенные объекты в `watch` не поддерживаются в v1
- Прокси не используется — дифф делается вручную движком

Порядок событий внутри `act`:

```
колбэк нажат
  → снять значения watch-ключей из state
  → выполнить колбэк (мутация state)
  → сравнить с новыми значениями
  → собрать строки уведомлений
  → ре-рендер сцены
  → показать очередь snackbar-ов
```

Отклонённые варианты:

| Вариант | Причина отклонения |
|---|---|
| Реактивный `Proxy` | При восстановлении из истории нужно переоборачивать plain object в Proxy; скрытый слой, ломается неожиданно |
| Императивный `notify()` | Бойлерплейт в каждом `act`; автор должен вручную запоминать старое значение |
| Автодифф всех ключей | Нет контроля над тем, что показывается; формат только дефолтный |
