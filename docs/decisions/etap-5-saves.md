# Решения: Этап 5 — Система сохранений и история

## Два независимых механизма

Этап вводит два отдельных механизма с общей структурой снимка (`Snapshot`), но разным хранилищем:

```
Snapshot = { sceneKey: string, state: unknown, sceneLocals: Record<string, Record<string, unknown>> | null }

История          (in-memory)     ← undo / redo, не переживает перезагрузку страницы
Слоты сохранений (localStorage)  ← именные + auto, переживают перезагрузку
```

Клонирование — везде `JSON.parse(JSON.stringify(...))`, как решено в `etap-2-state.md`.

---

## На каких событиях пишется история

История пишется **на каждом `goto` и `act`** (подход RenPy, не Twine).

Снимок берётся **до** мутации состояния — так, чтобы undo возвращал состояние «перед действием»:

```
act():
  [1] historyPush(clone of state + sceneLocals + currentSceneKey)  ← ДО мутации
  [2] cb(state)
  [3] rerender()

goto():
  [1] historyPush(clone of state + sceneLocals + currentSceneKey)  ← ДО мутации
  [2] cb(state)  (если передан)
  [3] autoSave → localStorage                                      ← после cb, ДО рендера
  [4] runGameLoop(newKey)
```

---

## Кольцевой буфер истории

- Реализуется в памяти как кольцевой буфер фиксированного размера N
- N конфигурируется через `GameConfig.saves.historySize`, значение по умолчанию — **20**
- Отдельный указатель текущей позиции в буфере (`ptr`) обеспечивает undo/redo без перемещения элементов
- При добавлении нового снимка redo-стек (вся история «правее» ptr) **очищается** — следующий undo шаг недоступен

Redo-стек сбрасывается при любом `goto` или `act`, кроме самого undo.

---

## Автосохранение

Автосохранение срабатывает **при каждом переходе между сценами** (`goto`), **до** вызова `runGameLoop`:

```ts
// Момент записи: после cb(state), до runGameLoop(newKey)
autoSave({
  sceneKey:    newKey,
  state:       jsonClone(currentState),
  sceneLocals: {},       // новая сцена ещё не загружена, локалей нет
})
```

При загрузке из автосохранения игрок оказывается в начале новой сцены — локали инициализируются заново при первом `ctx.local()`.

Автосохранение хранится в отдельном слоте с id `"auto"` и не занимает именные слоты.

---

## Именные слоты сохранений

### Количество слотов

Конфигурируется через `GameConfig.saves.slots`, значение по умолчанию — **5**.

```ts
defineGame({
  saves: { slots: 3, historySize: 30 },
  state,
  scenes,
})
```

### Структура `SaveSlot`

```ts
interface SaveSlot {
  id:          string   // 'auto' | 'slot-0' | 'slot-1' | ...
  savedAt:     string   // ISO timestamp
  label:       string   // текстовый thumbnail (только в именных слотах)
  sceneKey:    string   // ключ сцены для восстановления (не показывается игроку)
  state:       unknown  // JSON-клон состояния игры
  sceneLocals: Record<string, Record<string, unknown>> | null
}
```

### Thumbnail / label

- **Автосохранение**: `label` не используется — в UI отображается только дата и время
- **Именной слот**: `label = document.getElementById('scene-content')?.textContent?.slice(0, 80) ?? ''`
  Снимается в момент ручного сохранения — сцена уже отрисована

Ключ сцены (`sceneKey`) хранится внутри слота и используется только при восстановлении — игроку не показывается.

### Схема хранилища

```ts
// localStorage key: 'ifkit:saves'
interface SavesStore {
  auto:  SaveSlot | null
  slots: Array<SaveSlot | null>  // длина = GameConfig.saves.slots
}
```

## UI модального окна сохранений

Открывается по кнопке `#btn-saves` как модальное окно — по тому же паттерну, что `settings-modal.ts` (backdrop, focus trap, Escape/клик-за-модалкой).

```
┌────────────────────────────────────────────────────────┐
│ ✕                   Сохранения                         │
├────────────────────────────────────────────────────────┤
│ [Авто]  23.04.2026 14:30                       [Загр.] │
│         «Ты входишь в тёмный лес...»                   │
├────────────────────────────────────────────────────────┤
│ [  1 ]  22.04.2026 09:12              [Загр.] [Удалить]│
│         «Деревня встречает тебя приветливо...»          │
├────────────────────────────────────────────────────────┤
│ [  2 ]  ── пусто ──                         [Сохранить]│
├────────────────────────────────────────────────────────┤
│ [  3 ]  ── пусто ──                         [Сохранить]│
└────────────────────────────────────────────────────────┘
```

- Сохранение и загрузка — **в одном экране**
- Автосохранение не имеет кнопок «Сохранить» и «Удалить»
- Запись поверх занятого слота и удаление слота требуют **inline-подтверждения**:
  ```
  До:    [Загрузить]  [Удалить]
  После: [Загрузить]  Удалить? [Да] [Нет]
  ```
- ARIA: `role="dialog"`, `aria-modal="true"`, `aria-labelledby`

## Точка интеграции: `restoreGameState`

Для восстановления из снимка (undo, redo, load) в `scenes.ts` вводится функция:

```ts
export function restoreGameState(snapshot: Snapshot): void {
  currentState = jsonClone(snapshot.state)
  if (snapshot.sceneLocals) {
    for (const [key, locals] of Object.entries(snapshot.sceneLocals)) {
      restoreSceneLocal(key, locals)
    }
  }
  runGameLoop(snapshot.sceneKey)
}
```

`runGameLoop` при этом работает в обычном режиме: если `currentSceneKey !== snapshot.sceneKey`, он очистит локали прежней сцены — что корректно, потому что нужные локали уже восстановлены до вызова.

## Отклонённые варианты

| Вопрос | Отклонённый вариант | Причина |
|--------|---------------------|---------|
| Thumbnail | `html2canvas` / Canvas API | Нарушает принцип zero-dep и single-file |
| Thumbnail | CSS-snapshot | Не визуальный; сложнее без выгоды |
| История | Только на `goto` | Нельзя откатить действие внутри сцены (Twine-подход) |
| Автосохранение | После рендера | При загрузке textContent уже есть в state, render вызывается заново; до рендера — проще и надёжнее |
| UI сохранений | Раздельные экраны «Сохранить» / «Загрузить» | Лишний переход; один экран покрывает оба сценария |
| Удаление | `window.confirm()` | Нарушает стиль UI; inline-подтверждение согласовано с дизайном |
