# ifKit / Intaractive Fiction Kit

## Цель

Проект, который станет каркасом для написания интерактивной литературы с плавной кривой обучения от непрограммиста до опытного разработчика публикющегося в steam.

## Технический стек

- vite
- typrescript
- vite-plugin-singlefile
- tauri
- ограничение железа: проект должен собираться даже на андроид-смартфонах

## Архитектурные ограничения
- api проекта должно быть понятно даже непрограммисту
- проект должен легко расширяться

## Стиль кода 
- для записи сценария используем словарь с ключами типа строка и значением типа лямбда-функция
- для записи разметки используется Builder Callback Pattern

# Описание проекта

Проект, который станет каркасом для написания интерактивной литературы с плавной кривой обучения от непрограммиста до опытного разработчика публикющегося в steam.

Зона с действиями на сцене.

Зона с переходами со сцены.

## Дополнительная

Так же должны быть настройки и сохранения

# Описание api

Пользователь записывает сценарий игры в файл game.ts

Статичная часть это лямбда-функция, принимающая на вход состояние игры. Рендер производится при помощи теговых функций, записывающих состояние. То есть движок решает отрендерить статичную часть, очищает переменную в которую рендер-функции записывают результат, вызывает эту лямбда-функцию, в лямбда-функции вызываются теговые функции, теговые функции записывают результат своей работы в переменную, движок получает итоговый рендер из переменной. (Аналогично происходит управление звуком.) Рендер-функции, которые записывают в буфер начинаются с заглавной буквы, которые возвращают значения с маленькой (например P для абзаца и p для абзаца внутри div). Резултат работы и тех и других корректная html-вёрстка.

Сцены хранятся в объекте, ключи которого строки, а значения лямбда-функции. Сцены рендерятся так же как и статичная часть, с поправкой, что они ещё список действий и переходов составляют, но уже не через теговый функции, а обычные, причём функция перехода принимает только валидные названия сцен (typeof keyof объекта в котором сцены хранятся).

Для хранения состояния, которые нужно сохранять в при сохранении игры используется два объекта. Один пользователь заполняет сам, в нём хранятся глобальные переменные, в другой запись идёт через специальную функцию, чтобы пользователь мог добавить локальную переменную прямо внутри сцены. (или придумай другой способ хранения)

# Требования

Что должно быть:
- низкие системные требования
- подробная документация
- низкий порог входа
- расширяемость
- запоминание последних n состояний игры
- система сохранений с большим количеством слотов у которых есть не только номер с датой, но и описание/скриншот
- автоматическая интернационализация
- сборка как в один файл html так и в приложение desktop/mobile
- адаптивный и доступный дизайн (разные размеры экрана, скринридер)
- переходы с локации оформлены в виде формы с радиокнопками и кнопкой подтверждения перехода
- информирование об изменении явных переменных через snackbar
- мощный компилятор
- поддержка стандартными средствами разработки
- вывод звука (фоновая музыка, звуки)
- настройки размера шрифта
- настройки громкости звуков и музыки
- настройки темы

Чего не должно быть:
- сложный код
- высокого порога входа
- исчезающий художественный текст (должна быть возможность откатиться назад и перечитать)
